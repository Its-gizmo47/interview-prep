    01. Indexes & Composite Indexes

    SENIOR

    Explanation. An index is a separate, sorted data structure (typically a B-tree in MySQL/InnoDB) that lets the database find rows without scanning the entire table — trading extra storage and slower writes for much faster reads on the indexed column(s). A composite (multi-column) index is sorted by its columns in order, which matters enormously: it can efficiently satisfy a query filtering on the first column alone, or the first and second together, but generally can't be used efficiently to filter on the second column alone — this is the leftmost prefix rule. A covering index includes every column a query needs, letting the database satisfy the query from the index alone without touching the actual table rows.

    Why interviewers ask it

    Indexing is the highest-leverage, most practically-tested database skill because it's the difference between a query that takes milliseconds and one that takes seconds at scale — and misunderstanding the leftmost-prefix rule is an extremely common real bug where a composite index silently isn't being used at all.

    Real-world use case

    In a 127-table schema like CampVine's, a bookings table queried by both camp_id and a date range needs a composite index on (camp_id, date) in that order — a query filtering only by date wouldn't use it efficiently, but the actual access pattern (find bookings for a specific camp within a date range) hits it perfectly.

    Common mistakes
    • Indexing every column "just in case" — each index slows down every write (insert/update/delete) since the index structure has to be maintained too, so indexes are a deliberate trade-off, not free.
    • Creating a composite index in the wrong column order for the actual query pattern, making it unusable for the most common query.
    • Not knowing that a leading wildcard (LIKE '%term') can't use a standard B-tree index at all, while a trailing wildcard (LIKE 'term%') can.
    • Forgetting that functions applied to an indexed column in a WHERE clause (WHERE YEAR(created_at) = 2026) prevent the index from being used — the column needs to be compared directly.
    Interview answer
    "I index based on actual query patterns, not preemptively — every index speeds up reads but costs write performance and storage, so it's a deliberate trade-off. For composite indexes, column order has to match how the query actually filters, because of the leftmost prefix rule — an index on (camp_id, date) helps a query filtering by camp_id alone or camp_id plus date, but not date alone. When I suspect an index isn't being used, I check with EXPLAIN rather than guessing — a full table scan shows up clearly as type: ALL instead of a ref or range lookup."
    Follow-up questions
    • You have an index on (a, b) — will a query filtering only on b use it? Why or why not?
    • What's a covering index, and how would you design one for a specific slow query?
    • Why can indexing hurt write-heavy tables?
    -- Composite index matching the real access pattern
CREATE INDEX idx_camp_date ON bookings (camp_id, booking_date);

-- Uses the index (leftmost prefix: camp_id, or camp_id + booking_date)
SELECT * FROM bookings WHERE camp_id = 42 AND booking_date >= '2026-01-01';

-- Does NOT use idx_camp_date efficiently — booking_date isn't the leading column
SELECT * FROM bookings WHERE booking_date >= '2026-01-01';

-- Covering index: query is satisfied from the index alone, no table lookup
CREATE INDEX idx_covering ON bookings (camp_id, booking_date, status);
SELECT status FROM bookings WHERE camp_id = 42 AND booking_date >= '2026-01-01';
    Best practices
    • Index based on real, measured query patterns (slow query log) rather than speculatively.
    • Put the most selective / most-filtered-on column first in a composite index, matching actual WHERE clause usage.

    02. Normalization vs Denormalization

    MID

    Explanation. Normalization organizes data to eliminate redundancy by splitting it into related tables, following progressively stricter normal forms — 1NF (atomic columns, no repeating groups), 2NF (no partial dependency on part of a composite key), 3NF (no transitive dependencies — non-key columns depend only on the key). This prevents update anomalies (changing a customer's address in one place but not another) at the cost of needing joins to reassemble data. Denormalization deliberately reintroduces redundancy — duplicating or precomputing data — to avoid expensive joins/aggregations on read-heavy paths, accepting the risk of data getting out of sync in exchange for read speed.

    Why interviewers ask it

    This tests whether you understand database design as a set of trade-offs rather than a rulebook — a fully normalized schema is "textbook correct" but can be genuinely wrong for a read-heavy, high-traffic table, and knowing when to break the rules deliberately is a senior signal.

    Real-world use case

    Storing a camp's current average rating as a denormalized column (updated on each new review) instead of recalculating AVG(rating) from the reviews table on every single listing page load is a textbook denormalization trade-off — worth it for a frequently-read, infrequently-changed aggregate on a high-traffic page.

    Common mistakes
    • Denormalizing prematurely, before actually measuring that joins/aggregation are a real performance problem — adding synchronization complexity for no proven benefit.
    • Denormalizing without a clear strategy for keeping the duplicated data in sync (via triggers, application logic, or event listeners), leading to silently stale or inconsistent data.
    • Over-normalizing a schema to the point where every simple read requires five joins, when a few strategic redundant columns would be far simpler.
    Interview answer
    "I start normalized by default — it prevents update anomalies and keeps a single source of truth — and only denormalize deliberately, on a specific measured bottleneck, usually a read-heavy aggregate that's recalculated far more often than the underlying data changes. When I do denormalize, like caching a camp's average rating as a column, I make sure there's a clear, reliable mechanism to keep it in sync — updating it in the same transaction as the review write, or via a model observer — rather than hoping application code remembers to update it everywhere."
    Follow-up questions
    • Give an example of an update anomaly a normalized schema prevents.
    • How would you keep a denormalized aggregate column in sync reliably?
    • What's the difference between 2NF and 3NF, concretely?
    -- Normalized: rating computed on read, always accurate, costs a join+aggregate
SELECT c.name, AVG(r.rating) as avg_rating
FROM camps c JOIN reviews r ON r.camp_id = c.id
GROUP BY c.id;

-- Denormalized: precomputed column, cheap read, needs a sync mechanism
ALTER TABLE camps ADD COLUMN avg_rating DECIMAL(3,2) DEFAULT 0;
-- kept in sync via a model observer or DB trigger whenever a review is created/updated/deleted
    Best practices
    • Normalize first; denormalize only against a measured, real bottleneck.
    • Whenever you denormalize, decide and document exactly what keeps the redundant data in sync.

    03. Joins — Inner, Left, Right, Cross, Self

    MID

    Explanation. INNER JOIN returns only rows with matches in both tables. LEFT JOIN returns every row from the left table, with NULLs filling in for unmatched right-table columns — used whenever you need "all of X, plus related Y if it exists" (e.g. all camps, even ones with zero bookings). RIGHT JOIN is the mirror image, rarely used in practice since any right join can be rewritten as a left join by swapping table order. CROSS JOIN produces the full Cartesian product — every row of table A paired with every row of table B — genuinely useful for generating combinations (e.g. every date × every camp for an availability calendar) but a common accidental bug when a join condition is forgotten. SELF JOIN joins a table to itself, used for hierarchical or comparative data within the same table (e.g. an employee and their manager, both rows in the same employees table).

    Why interviewers ask it

    Join type selection is tested constantly because picking the wrong one is a very common, subtle bug — using an INNER JOIN when you actually needed a LEFT JOIN silently drops rows (e.g. camps with no bookings disappear from a report entirely) without throwing any error.

    Real-world use case

    A dashboard showing "all camps and their booking count, including camps with zero bookings" needs a LEFT JOIN with COUNT — an INNER JOIN would silently exclude camps that haven't been booked yet, which is exactly the kind of report bug that goes unnoticed until someone asks why a known camp is missing.

    Common mistakes
    • Defaulting to INNER JOIN out of habit when the requirement is actually "include rows with no match" — silently drops data instead of erroring, making it a hard bug to catch.
    • Forgetting the join condition and accidentally producing a Cartesian product (an unintentional cross join), which can silently multiply row counts and break aggregates.
    • Filtering a LEFT JOIN's right-table column in the WHERE clause instead of the ON clause, which effectively turns it back into an INNER JOIN by excluding the NULL rows the LEFT JOIN was meant to preserve.
    Interview answer
    "The join type is really a question about what happens to unmatched rows. INNER JOIN drops them from both sides, LEFT JOIN keeps every row from the left table and fills unmatched right-side columns with NULL — which is what I reach for whenever the requirement includes 'even if there's no related record,' like showing every camp including ones with zero bookings. The subtle bug I watch for is filtering a LEFT JOIN's right-table column in the WHERE clause instead of the ON clause — that silently turns it back into an INNER JOIN by excluding all the NULL rows you just went out of your way to keep."
    Follow-up questions
    • Write a query to find all camps with zero bookings, using a LEFT JOIN.
    • Why does filtering a LEFT JOIN's right table in WHERE instead of ON change the result?
    • Give a real use case for a self join.
    -- Camps with zero bookings — the classic LEFT JOIN + NULL check pattern
SELECT c.name
FROM camps c
LEFT JOIN bookings b ON b.camp_id = c.id
WHERE b.id IS NULL;

-- Self join: employee paired with their manager, same table
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- The WHERE-vs-ON bug: this silently becomes an INNER JOIN
SELECT c.name FROM camps c
LEFT JOIN bookings b ON b.camp_id = c.id
WHERE b.status = 'confirmed'; -- drops all camps with NO bookings at all
    Best practices
    • Put right-table filter conditions in the ON clause when you need to preserve a LEFT JOIN's unmatched rows.
    • Always double check row counts after a join during development — an unexpectedly inflated count is the classic sign of an accidental Cartesian product.

    04. Transactions, ACID & Isolation Levels

    SENIOR

    Explanation. A transaction groups multiple operations so they all succeed together or all roll back together — critical for anything with multiple related writes, like creating a booking and deducting inventory in the same operation. ACID defines the guarantees: Atomicity (all-or-nothing), Consistency (the database moves between valid states, respecting constraints), Isolation (concurrent transactions don't interfere with each other's intermediate state), Durability (once committed, data survives a crash). Isolation levels tune the trade-off between consistency and concurrency performance: Read Uncommitted (can see other transactions' uncommitted changes — "dirty reads"), Read Committed (only sees committed data, but can get different results on repeated reads within the same transaction), Repeatable Read (MySQL's default — consistent reads throughout the transaction, but can still see new rows appear — "phantom reads"), Serializable (strongest, fully isolated, but effectively serializes concurrent access, hurting throughput).

    Why interviewers ask it

    Any payment or booking system lives or dies on correct transaction handling — this is directly the difference between a reliable system and one that double-charges customers or double-books inventory under concurrent load, so it's a near-mandatory question for anyone claiming payment integration experience.

    Real-world use case

    Creating a booking and processing its payment need to be one atomic transaction — if the payment succeeds but the booking record write fails (or vice versa), you either have a paid-for booking that doesn't exist or a booking with no payment behind it. Wrapping both in a transaction, and only committing once both succeed, is exactly the kind of correctness guarantee that matters for the payment integration work you've done.

    Common mistakes
    • Doing a multi-step write (check availability, then insert booking) without a transaction or row locking, allowing two concurrent requests to both pass the availability check and double-book the same slot — a classic race condition.
    • Wrapping unrelated, independent operations in a single large transaction, holding locks longer than necessary and hurting concurrency.
    • Not knowing the default isolation level of the database in use, and assuming stronger guarantees (like no phantom reads) than what's actually configured.
    Interview answer
    "Any time I have multiple writes that need to succeed or fail together — like a booking record plus a payment charge — I wrap them in a transaction so a partial failure can't leave the data in an inconsistent state. For race conditions specifically, like two people trying to book the last spot at the same time, a transaction alone isn't enough — I also need row-level locking, either a pessimistic lock with SELECT ... FOR UPDATE, or an optimistic approach with a version column, depending on how much contention I expect. I keep transactions as short and focused as possible, since they hold locks and hurt concurrency the longer they're open."
    Follow-up questions
    • What's a dirty read vs a phantom read? Which isolation levels prevent each?
    • How would you prevent two users from double-booking the last available slot?
    • What's the difference between optimistic and pessimistic locking, and when would you use each?
    -- Atomic booking + payment: both succeed or both roll back
START TRANSACTION;
  INSERT INTO bookings (camp_id, user_id, amount) VALUES (42, 7, 400);
  UPDATE camp_slots SET available = available - 1 WHERE camp_id = 42 AND available > 0;
  -- if the UPDATE affected 0 rows (no availability), roll back instead of committing
COMMIT;

-- Pessimistic lock to prevent a double-booking race condition
START TRANSACTION;
  SELECT available FROM camp_slots WHERE camp_id = 42 FOR UPDATE; -- blocks other transactions until commit
  -- ... check availability in application code, then insert booking
COMMIT;
    Best practices
    • Keep transactions short-lived — do validation/preparation outside the transaction where possible, minimizing lock duration.
    • Use row-level locking (or optimistic concurrency with a version column) explicitly whenever a race condition between concurrent writers is a realistic risk.

    05. Query Optimization & EXPLAIN

    SENIOR

    Explanation. EXPLAIN shows the database's query execution plan without actually running the query — which indexes (if any) it plans to use, the estimated number of rows it'll scan, the join order, and the access type (a full table scan vs an index lookup). The type column is the key signal: ALL means a full table scan (worst case), index scans the whole index, range/ref use an index for a bounded lookup (good), and const/eq_ref mean at most one matching row (best case). Beyond indexing, optimization also covers avoiding SELECT * (fetches unneeded columns, prevents covering-index optimization), rewriting correlated subqueries as joins where possible, and batching operations instead of looping queries in application code.

    Why interviewers ask it

    This is the practical, hands-on skill behind "the query is slow, fix it" — a question type nearly every backend interview includes in some form, since diagnosing real performance problems (not just knowing index theory) is what the job actually requires day to day.

    Real-world use case

    Diagnosing a slow endpoint in production starts with running EXPLAIN on the query the endpoint generates (visible via Laravel's query log or Debugbar), checking whether type shows a full table scan, and then either adding the missing index, rewriting the query, or restructuring the eager-loading to eliminate an N+1 hiding behind it.

    Common mistakes
    • Adding an index reactively without confirming via EXPLAIN that it actually gets used — sometimes MySQL's query planner chooses not to use an index that exists, for reasons worth understanding (e.g. low table cardinality, or the optimizer estimating a full scan is actually cheaper).
    • Optimizing by intuition instead of measurement — guessing which query is slow rather than checking the slow query log or actual EXPLAIN output.
    • Ignoring that ORDER BY and GROUP BY can also benefit from (or require) an index to avoid an expensive in-memory sort ("Using filesort" in EXPLAIN output).
    Interview answer
    "My first step diagnosing a slow query is always EXPLAIN before touching anything — I look at the type column to see if it's doing a full table scan versus an index lookup, and at rows to see how many rows it estimates scanning. From there it's usually one of a few fixes: add or adjust an index to match the actual filter/join/order pattern, rewrite a correlated subquery as a join, or realize the 'slow query' is actually N+1 queries disguised as one slow page load. I don't add indexes speculatively — I confirm with EXPLAIN that the fix actually changes the access type before considering it solved."
    Follow-up questions
    • What does 'Using filesort' in an EXPLAIN output mean, and how would you fix it?
    • Why might MySQL choose not to use an index that exists on a column you're filtering by?
    • Walk through your process for diagnosing a slow endpoint in production.
    EXPLAIN SELECT * FROM bookings WHERE camp_id = 42 AND booking_date >= '2026-01-01';

-- Key columns to read in the output:
-- type: ALL (full scan, bad) vs range/ref (index used, good) vs const (best)
-- key:  which index (if any) MySQL actually chose to use
-- rows: estimated rows scanned to satisfy the query
-- Extra: "Using filesort" / "Using temporary" are red flags for optimization

-- Avoid SELECT * when only specific columns are needed —
-- lets a covering index satisfy the query without touching table rows
SELECT id, status FROM bookings WHERE camp_id = 42;
    Best practices
    • Always confirm an optimization actually changed the query plan via EXPLAIN — don't assume.
    • Select only the columns actually needed, especially on large or frequently-run queries.

    06. Subqueries vs Joins vs CTEs

    MID

    Explanation. A subquery is a query nested inside another — it can appear in SELECT, FROM, or WHERE. A correlated subquery references a column from the outer query and re-runs once per outer row, which can be genuinely slow at scale; an uncorrelated subquery runs once independently. A Common Table Expression (CTE), defined with WITH ... AS (...), names a temporary result set for readability and reuse within a single query — functionally similar to a subquery in FROM, but easier to read, and required for recursive queries (e.g. walking a category tree or org chart) via WITH RECURSIVE. In many cases a correlated subquery can be rewritten as a join, which the query planner can often optimize better.

    Why interviewers ask it

    Rewriting a slow correlated subquery as a join is one of the most common concrete query-optimization tasks in practice, and CTEs specifically test whether you know modern SQL beyond basic joins — increasingly expected as codebases move complex reporting logic into the database layer.

    Real-world use case

    A recursive CTE is the natural tool for a hierarchical structure like camp categories with subcategories, or an org chart of camp staff roles — walking the full tree in one query instead of recursive application-level queries per level.

    Common mistakes
    • Using a correlated subquery inside a SELECT clause across a large result set when a single joined aggregate query would compute the same thing far more efficiently.
    • Not knowing CTEs (non-recursive) are often just syntactic sugar for readability — the query planner may or may not materialize them separately depending on the database engine and version, so a CTE isn't automatically a performance win over a subquery.
    • Writing a recursive CTE with no proper termination condition, risking an infinite loop on cyclic data.
    Interview answer
    "I reach for a CTE mainly for readability — naming an intermediate result set makes a complex query much easier to follow than nesting subqueries three levels deep — and it's the only real option once you need recursion, like walking a category tree. For performance specifically, though, I check whether a correlated subquery can be rewritten as a join, since the query planner often handles that better than re-running a subquery once per outer row; I confirm with EXPLAIN rather than assuming one form is always faster than the other."
    Follow-up questions
    • Rewrite a correlated subquery in WHERE as an equivalent join.
    • When would you need WITH RECURSIVE, and what stops it from looping forever?
    • Is a CTE always faster than an equivalent subquery? Why or why not?
    -- Correlated subquery — re-evaluated per outer row
SELECT name FROM camps c
WHERE (SELECT COUNT(*) FROM bookings b WHERE b.camp_id = c.id) > 10;

-- Rewritten as a join + GROUP BY/HAVING — usually the more optimizable form
SELECT c.name FROM camps c
JOIN bookings b ON b.camp_id = c.id
GROUP BY c.id HAVING COUNT(*) > 10;

-- Recursive CTE walking a category tree
WITH RECURSIVE category_tree AS (
  SELECT id, name, parent_id FROM categories WHERE parent_id IS NULL
  UNION ALL
  SELECT c.id, c.name, c.parent_id FROM categories c
  JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree;
    Best practices
    • Try rewriting slow correlated subqueries as joins first, and verify the change with EXPLAIN.
    • Use CTEs to make complex, multi-step queries readable and maintainable, not just for perceived performance.

    07. Views, Stored Procedures & Triggers

    MID

    Explanation. A view is a saved, named query that behaves like a virtual table — it doesn't store data itself (unless it's a materialized view, less common in MySQL), it just wraps a query for reuse and to present a simplified or restricted interface over complex underlying tables. A stored procedure is a precompiled block of SQL logic stored in the database itself, callable by name, useful for encapsulating multi-step database operations. A trigger is a piece of logic that fires automatically in response to a table event (BEFORE/AFTER INSERT/UPDATE/DELETE) — commonly used for auditing, enforcing invariants, or keeping a denormalized column in sync without relying on application code to remember.

    Why interviewers ask it

    This topic tests judgment about where logic should live — database-level constructs are powerful but can also hide business logic from the application layer, making it invisible to code review, version control (if not managed carefully), and harder to test — so interviewers want to see you weigh that trade-off, not just know the syntax.

    Real-world use case

    A trigger that automatically updates a camp's avg_rating column whenever a row is inserted/updated/deleted in the reviews table is a legitimate use — it guarantees consistency regardless of which application code path writes a review, which a Laravel model observer can't guarantee if the row is ever modified outside of Eloquent (a raw migration, a manual DB fix, another service).

    Common mistakes
    • Putting substantial business logic in stored procedures/triggers in a codebase that otherwise keeps logic in the application layer — creates two places to look for behavior, and triggers especially can surprise developers who don't expect a simple UPDATE to have side effects.
    • Relying on a Laravel model observer alone to maintain a denormalized column's consistency, forgetting that any write that bypasses Eloquent (raw SQL, another service, a migration) won't trigger it — a DB-level trigger doesn't have that gap.
    • Not version-controlling database-level logic (procedures/triggers) alongside application code, making changes to it invisible in code review.
    Interview answer
    "I generally prefer keeping business logic in the application layer where it's version-controlled, testable, and visible in code review — but there are cases where a database-level guarantee is genuinely stronger than an application-level one, like a trigger keeping a denormalized column in sync regardless of which code path writes to the underlying table, including raw SQL or another service entirely. When I do use triggers or procedures, I keep them narrowly scoped to that kind of invariant-enforcement or sync logic, not general business rules, and make sure they're captured in migrations so they're version-controlled like everything else."
    Follow-up questions
    • What's a real risk of putting business logic in a trigger instead of application code?
    • When would a database-level guarantee (trigger) be genuinely better than an application-level one (model observer)?
    • What's the difference between a regular view and a materialized view?
    -- View: simplified interface over a join, reusable
CREATE VIEW active_camp_bookings AS
SELECT b.id, c.name AS camp_name, b.booking_date
FROM bookings b JOIN camps c ON c.id = b.camp_id
WHERE c.status = 'active';

-- Trigger: keeps avg_rating in sync regardless of write source
CREATE TRIGGER after_review_insert
AFTER INSERT ON reviews
FOR EACH ROW
UPDATE camps
SET avg_rating = (SELECT AVG(rating) FROM reviews WHERE camp_id = NEW.camp_id)
WHERE id = NEW.camp_id;
    Best practices
    • Default to application-layer logic for anything testable and business-rule-related; reserve triggers for narrow, unavoidable consistency guarantees.
    • Version-control database-level logic through migrations, same as schema changes.

    08. Keys & Constraints: PK, FK, Unique, Referential Integrity

    MID

    Explanation. A primary key uniquely identifies each row and is non-null by definition; most engines automatically index it. A foreign key enforces referential integrity — it guarantees a value in one table must exist as a primary key in another, preventing orphaned records (a booking pointing at a deleted camp). Foreign keys carry an ON DELETE/ON UPDATE behavior: CASCADE (delete/update dependent rows automatically), RESTRICT (block the operation if dependents exist), SET NULL (null out the reference). A unique constraint guarantees no duplicate values in a column (or column combination) but, unlike a primary key, allows NULL and a table can have multiple unique constraints.

    Why interviewers ask it

    Constraints are the database's own enforcement of data integrity, independent of whatever application code happens to run — this topic tests whether you rely on the database as a genuine safety net or only on application-level validation, which can be bypassed or buggy.

    Real-world use case

    A composite unique constraint on (camp_id, user_id, booking_date) prevents the same user from double-booking the identical slot at the database level, even if application-level validation has a bug or race condition — a real defense-in-depth decision relevant to exactly the kind of concurrent booking system you've built.

    Common mistakes
    • Relying solely on application-level validation for uniqueness/integrity, which is vulnerable to race conditions between the check and the write — a database constraint closes that gap because it's enforced atomically at write time.
    • Using ON DELETE CASCADE without fully considering the blast radius — accidentally deleting a camp cascades to delete every booking, payment record, and review tied to it.
    • Not adding a unique constraint on a column that's logically unique (like an email or an idempotency key) and finding duplicates only after they've already been created in production.
    Interview answer
    "I treat database constraints as the actual source of truth for data integrity, with application-level validation as a fast, friendly first line of defense — not the only one — because application checks are vulnerable to race conditions that a database-level unique constraint closes atomically. For foreign key delete behavior I'm deliberate rather than defaulting to CASCADE everywhere — for something like a camp with bookings and payment history attached, I'd rather RESTRICT the delete and force an explicit decision than silently cascade-delete financial records."
    Follow-up questions
    • Why isn't application-level uniqueness validation alone sufficient to prevent duplicates?
    • When would ON DELETE CASCADE be dangerous, and what would you use instead?
    • What's the difference between a unique constraint and a unique index?
    CREATE TABLE bookings (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  camp_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  booking_date DATE NOT NULL,
  FOREIGN KEY (camp_id) REFERENCES camps(id) ON DELETE RESTRICT,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  UNIQUE KEY uq_no_double_booking (camp_id, user_id, booking_date)
);
    Best practices
    • Enforce true business-level uniqueness with a database constraint, not just application validation.
    • Choose ON DELETE behavior deliberately per relationship — never let CASCADE be the unconsidered default on financially or historically significant data.

    09. Aggregation: GROUP BY, HAVING & Window Functions

    SENIOR

    Explanation. GROUP BY collapses rows sharing a value into a single summary row per group, used with aggregate functions (COUNT, SUM, AVG, MAX, MIN). WHERE filters rows before grouping; HAVING filters groups after aggregation — which is why you can't reference an aggregate function's result in WHERE, only in HAVING. Window functions (ROW_NUMBER(), RANK(), SUM() OVER (...)) are the more powerful modern tool: unlike GROUP BY, they compute an aggregate without collapsing rows — each row keeps its own identity while also seeing an aggregate calculated over a defined "window" of related rows (via PARTITION BY and ORDER BY), enabling things like "rank each booking within its camp by date" or a running total, which are awkward or impossible with plain GROUP BY.

    Why interviewers ask it

    Window functions specifically are a strong differentiator — many developers only know GROUP BY and reach for slow, awkward self-joins or application-level loops for things a window function solves in one clean query, so demonstrating fluency here signals real SQL depth beyond CRUD basics.

    Real-world use case

    Finding each camp's most recent booking, or ranking camps by revenue within each region, is exactly what ROW_NUMBER() OVER (PARTITION BY ...) is built for — computing a per-group rank while still returning individual row detail, something a plain GROUP BY can't do since it would collapse the very rows you need to see.

    Common mistakes
    • Trying to filter on an aggregate in WHERE instead of HAVING — a common syntax-level mistake that also reveals a misunderstanding of query execution order.
    • Reaching for a self-join or an application-level loop to compute a running total or per-group rank when a window function does it in one pass, more efficiently and more readably.
    • Forgetting PARTITION BY in a window function when a per-group (not global) calculation was intended, accidentally computing the aggregate across the entire result set.
    Interview answer
    "GROUP BY collapses rows into one summary per group, so it's right when I only need the aggregate itself — total bookings per camp. The moment I need the aggregate alongside individual row detail — like each booking's rank within its camp, or a running total — GROUP BY can't do that without losing row-level data, so I reach for a window function with PARTITION BY to scope the calculation per group while keeping every row intact. And WHERE vs HAVING is really about execution order: WHERE filters rows before grouping happens, HAVING filters the resulting groups, which is why you can't put an aggregate condition in WHERE."
    Follow-up questions
    • Write a query to find the most recent booking per camp using a window function.
    • What's the difference between RANK() and ROW_NUMBER() when there are ties?
    • Why can't you filter on COUNT(*) in a WHERE clause?
    -- GROUP BY + HAVING: total bookings per camp, only camps with 10+
SELECT camp_id, COUNT(*) AS total
FROM bookings
GROUP BY camp_id
HAVING COUNT(*) >= 10;

-- Window function: most recent booking per camp, WITHOUT collapsing rows
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY camp_id ORDER BY booking_date DESC) AS rn
  FROM bookings
) ranked
WHERE rn = 1;

-- Running total per camp, ordered by date
SELECT camp_id, booking_date, amount,
  SUM(amount) OVER (PARTITION BY camp_id ORDER BY booking_date) AS running_total
FROM bookings;
    Best practices
    • Reach for window functions before self-joins or app-level loops for per-group ranking, running totals, or "latest per group" queries.
    • Remember the logical execution order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY — it explains most GROUP BY/HAVING confusion.

    10. Pagination & Scaling: Offset vs Cursor, Replication, Sharding

    SENIOR

    Explanation. Offset-based pagination (LIMIT 20 OFFSET 100) is simple but gets progressively slower on deep pages, because the database still has to scan and discard all the skipped rows before returning the page — and it's unstable if rows are inserted/deleted between page loads, causing skipped or duplicated rows. Cursor-based (keyset) pagination instead uses the last row's sort value as a bookmark (WHERE id > :last_id ORDER BY id LIMIT 20), which stays fast at any depth since it goes straight to an index position instead of scanning and discarding. For scaling beyond a single database: replication keeps read replicas in sync with a primary (writes go to the primary, reads can be distributed across replicas) for read scaling and redundancy; sharding splits data horizontally across multiple database instances by some key (e.g. by camp_id range or hash) when a single instance can't hold or serve the full dataset — a much bigger architectural commitment than replication, since cross-shard queries and joins become significantly harder.

    Why interviewers ask it

    Offset pagination degrading at scale is a very common real production issue that only shows up once a table has genuinely large row counts — asking about it tests whether you've actually operated something at scale versus only built features against small dev datasets.

    Real-world use case

    A camp listing/search results feed that supports infinite scroll is exactly where cursor-based pagination earns its complexity over offset — as the dataset grows, offset pagination on page 500 would mean scanning and discarding 10,000 rows before returning the next 20, while cursor pagination stays consistently fast regardless of depth.

    Common mistakes
    • Defaulting to offset pagination everywhere without recognizing it degrades on large tables/deep pages — fine for a small admin table, a real problem for a public search feed at scale.
    • Implementing cursor pagination without a properly indexed, unique sort column (or tuple) to build the cursor from, which breaks the "stays fast at any depth" guarantee.
    • Reaching for sharding as a first solution to a scaling problem that read replication (or better indexing/caching) would have solved far more simply — sharding is a significant, often premature architectural commitment.
    Interview answer
    "For most admin-facing or shallow-paginated UIs, offset pagination is simple and fine. The moment there's a deep, high-traffic feed — public search results, infinite scroll — I switch to cursor-based pagination, keying off a unique, indexed sort column, since it stays fast at any depth instead of degrading as the offset grows. On scaling more broadly, I'd reach for read replicas long before sharding — replication is a much smaller architectural change and solves read-heavy load on its own; sharding is a last resort once a single instance genuinely can't hold or serve the dataset, because it makes cross-shard queries and joins significantly harder."
    Follow-up questions
    • Why does OFFSET get slower as the page number increases?
    • Implement cursor-based pagination for a feed sorted by created_at.
    • What's the difference between replication and sharding, and which would you reach for first?
    -- Offset pagination: gets slower deeper in — scans + discards 10,000 rows
SELECT * FROM camps ORDER BY id LIMIT 20 OFFSET 10000;

-- Cursor pagination: goes straight to the index position, stays fast
SELECT * FROM camps
WHERE id > 10000  -- :last_seen_id from the previous page
ORDER BY id
LIMIT 20;
    Best practices
    • Use cursor-based pagination for any deep, high-traffic, or infinite-scroll feed; keep offset for small/shallow admin views where simplicity wins.
    • Exhaust indexing, caching, and read replication before considering sharding — it's a major, hard-to-reverse architectural commitment.
