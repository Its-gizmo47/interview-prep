    01. The System Design Interview Framework

    SENIOR

    Explanation. System design interviews are deliberately open-ended and ambiguous — the actual thing being evaluated is less "did you arrive at the right architecture" and more "how do you think, and can you communicate a complex design out loud." A structured approach prevents the two most common failure modes: diving into details too early (spending 20 minutes on database schema before establishing what the system even needs to do), or staying so high-level the conversation never gets concrete enough to demonstrate real depth.

    1. 1. Clarify Requirements (5 min) Functional (what must it do — post a message, shorten a URL) and non-functional (scale, latency, consistency needs). Explicitly ask about scale — it changes the entire design.
    2. 2. Estimate Scale (3-5 min) Back-of-envelope: daily active users, requests/sec, read:write ratio, storage growth per year. These numbers directly justify later decisions (e.g. "at this write volume, a single DB instance won't hold up").
    3. 3. High-Level Design (10 min) Draw the major components and how data flows between them — client, load balancer, API servers, cache, database, queue — before drilling into any one of them.
    4. 4. Deep Dive (15-20 min) Pick 1-2 components the interviewer steers toward (or that are genuinely the interesting/hard part) and go deep — schema, algorithm, specific trade-off.
    5. 5. Identify Bottlenecks & Trade-offs (5 min) Where does this design break under 10x scale? What did you deliberately sacrifice (strong consistency for availability, cost for latency) and why.
    Why interviewers ask it

    At the level you're targeting, the interviewer already knows there's no single "correct" answer — they're evaluating whether you can navigate genuine ambiguity methodically, the same way you'd need to on a real, underspecified feature request from a product manager.

    Real-world use case

    This is exactly the muscle used when starting the CampVine DDD refactor with an ambiguous mandate as sole developer — clarifying what actually needed to change, estimating the real scope (127 tables), sketching bounded contexts at a high level before diving into any single domain's implementation.

    Common mistakes
    • Jumping straight to a detailed database schema or specific technology choice before establishing scale and requirements — the "right" answer for 1,000 users and 100 million users can be completely different designs.
    • Never asking clarifying questions at all, silently assuming requirements and building the wrong thing confidently.
    • Going so broad and generic ("we'd use a load balancer, then a database, then caching") that nothing demonstrates actual depth or a real decision being made.
    Interview answer
    "I start by nailing down functional and non-functional requirements explicitly, especially scale — because the right design for a thousand users and a hundred million users often isn't the same design at all. Then a quick back-of-envelope estimate — requests per second, read-write ratio, storage growth — since those numbers directly justify decisions I'll make later, like whether a single database can handle the write volume. Then I sketch the high-level components before drilling into any one deeply, and I explicitly let the interviewer's interest (or the genuinely hard part of the problem) steer where we go deep, rather than trying to design everything to the same level of detail in the time available."
    Follow-up questions
    • Walk me through your first two minutes on a completely new system design prompt.
    • Why does clarifying scale early change the rest of the design?
    • How do you decide which component to deep-dive into when the interviewer doesn't specify?
    // Back-of-envelope estimation — the numbers that justify later decisions
// Example: a camp booking search feature

Daily Active Users:        500,000
Avg searches per user/day: 4
Total requests/day:        2,000,000
Requests/sec (avg):        2,000,000 / 86,400 ≈ 23 req/s
Requests/sec (peak, 5x):   ≈ 115 req/s
Read:Write ratio:          ~20:1 (searches vastly outnumber bookings)

// 115 req/s is comfortably within a single well-indexed DB + cache layer —
// this number is WHY the design doesn't need sharding on day one.
    Best practices
    • Always state assumptions out loud rather than silently picking one — it invites correction and shows deliberate thinking.
    • Keep a rough time budget in mind so you reach the deep-dive phase — that's usually where the real signal is.

    02. Load Balancers & Reverse Proxies

    MID

    Explanation. A load balancer distributes incoming requests across multiple server instances so no single one is overwhelmed, and removes unhealthy instances from rotation via health checks — a prerequisite for horizontal scaling. Common algorithms: round robin (cycle through servers evenly), least connections (send to whichever server currently has the fewest active connections — better when requests vary in cost), and IP hash (route the same client consistently to the same server, needed for sticky sessions). A reverse proxy (Nginx, often the same piece of infrastructure as the load balancer in practice) sits in front of application servers, handling SSL termination, static file serving, request buffering, and hiding backend server details from clients entirely.

    Why interviewers ask it

    Load balancing is the first real infrastructure decision in almost every system design answer once a single server won't cut it — this tests whether you understand it's not just "add more servers" but requires an actual distribution strategy and health-checking mechanism.

    Real-world use case

    A booking platform's API servers sit behind a load balancer using least-connections (since a search request and a payment-processing request have very different costs), with health checks pulling an instance out of rotation automatically if it starts failing — preventing user-facing errors from one bad instance while the rest keep serving traffic.

    Common mistakes
    • Not accounting for the load balancer itself becoming a single point of failure — production setups typically need redundant load balancers, not just redundant app servers behind one.
    • Using sticky sessions (IP hash) by default when the backend is actually stateless — unnecessarily limits load distribution flexibility for no real benefit.
    • Forgetting health checks entirely, letting the load balancer keep routing traffic to an instance that's actually down or degraded.
    Interview answer
    "Once a single server can't handle the load, a load balancer is the first piece of infrastructure I add — it needs both a distribution algorithm, usually least-connections when request cost varies, and active health checks so a failing instance gets pulled out of rotation automatically rather than continuing to receive traffic. I also make sure the load balancer itself isn't a single point of failure — that means redundant load balancers in front of the server pool, not just redundant servers behind a single one."
    Follow-up questions
    • What's the difference between round robin and least connections, and when would you pick each?
    • How do you avoid the load balancer itself becoming a single point of failure?
    • Why might you need sticky sessions, and what's the trade-off?
    Client
  │
  ▼
┌─────────────────┐
│  Load Balancer   │  ← health checks each instance, routes around failures
└─────────────────┘
  │        │        │

│App 1│ │App 2│ │App 3│   ← stateless, any instance can serve any request

    Best practices
    • Keep application servers stateless so any instance can serve any request — this is what makes load balancing actually work cleanly.
    • Run redundant load balancers; a single one is just a relocated single point of failure.

    03. Horizontal vs Vertical Scaling & Statelessness

    MID

    Explanation. Vertical scaling (scaling up) means adding more resources — CPU, RAM — to a single machine; simple, no architectural change needed, but has a hard ceiling (the biggest machine available) and creates a single point of failure. Horizontal scaling (scaling out) means adding more machines and distributing load across them; a much higher ceiling and built-in redundancy, but requires the application to be designed for it — critically, requires the server itself to be stateless. A stateless server keeps no request-specific data in its own memory between requests (no in-memory session storage, no local file writes assumed to persist) — any server instance must be able to handle any request, with actual state living in a shared store (database, Redis) that every instance can reach.

    Why interviewers ask it

    This is the foundational trade-off underlying nearly every other scaling decision in system design — and statelessness specifically is the prerequisite that's easy to overlook until it actively breaks a horizontal scaling plan (e.g. in-memory session data that only exists on the one server instance a user happened to hit first).

    Real-world use case

    Storing user session data in each Node process's local memory works fine with one server, but breaks the moment there are two — a user's second request might land on a different instance with no memory of their session at all, which is exactly why moving session state to Redis (a shared store every instance can read) is a prerequisite for horizontally scaling that service, not an optional nice-to-have.

    Common mistakes
    • Assuming horizontal scaling is a drop-in infrastructure change, missing that it requires the application itself to already be stateless — this is an architectural decision, not just an ops one.
    • Vertically scaling indefinitely as the default response to load, hitting a hard ceiling later and requiring a much more disruptive migration to a horizontally-scaled architecture under pressure, instead of building statelessness in from the start.
    • Storing file uploads on local disk in a horizontally-scaled setup, where a file saved on one instance isn't visible to requests served by a different instance — needs shared/object storage (like S3) instead.
    Interview answer
    "Vertical scaling is simpler short-term but has a hard ceiling and no redundancy — you're still one machine failure away from an outage. Horizontal scaling has a much higher ceiling and built-in redundancy, but only works if the application is actually stateless — any instance needs to be able to handle any request, which means session data, uploaded files, anything request-specific has to live in a shared store like Redis or object storage, not in that instance's local memory or disk. I design for statelessness from the start even at small scale, since retrofitting it onto an app that's accumulated local state assumptions is a much bigger migration under pressure than building it in from day one."
    Follow-up questions
    • What specifically breaks if you horizontally scale a server that stores session data in local memory?
    • What's the ceiling problem with vertical scaling, concretely?
    • Where would you store user-uploaded files in a horizontally-scaled architecture?
    // Breaks under horizontal scaling — session lives only in THIS instance's memory
const sessions = {}; // in-memory object, unique per server process
app.post('/login', (req, res) => { sessions[userId] = { loggedIn: true }; });

// Works at any scale — session lives in a shared store every instance can reach
await redis.set(`session:${userId}`, JSON.stringify({ loggedIn: true }), 'EX', 3600);
    Best practices
    • Design for statelessness from the start, even before horizontal scaling is an immediate need — it's far cheaper than retrofitting.
    • Move any request-specific state (sessions, uploaded files) to a shared store reachable by every instance.

    04. CAP Theorem & Consistency Trade-offs

    SENIOR

    Explanation. CAP theorem states that a distributed system can only guarantee two of three properties simultaneously during a network partition (a real, eventually-inevitable failure where nodes can't communicate): Consistency (every read gets the most recent write, or an error), Availability (every request gets a response, though not necessarily the latest data), Partition tolerance (the system keeps working despite network partitions). Since partitions are a fact of distributed systems life (network failures happen), the real-world choice is actually between CP (consistent but may reject requests during a partition) and AP (available but may serve stale data during a partition) — not a free choice of any two. In practice, most systems don't apply one choice uniformly — different parts of the same application pick differently based on what that specific data needs (payment records: CP; view counts: AP).

    Why interviewers ask it

    CAP theorem tests whether you understand that distributed systems design is fundamentally about trade-offs, not a search for a design with no downsides — and specifically whether you can apply it concretely rather than reciting the definition without understanding what it means for an actual feature.

    Real-world use case

    A booking platform's payment/inventory data (has this slot been booked or not) needs to lean CP — it's better to briefly reject a booking request during a partition than to let two instances both confirm the same slot based on stale data — while something like a camp's view count or a "currently browsing" indicator can lean AP, since briefly showing a slightly stale number is a completely acceptable trade-off for staying available.

    Common mistakes
    • Treating CAP as "pick any 2 of 3" freely — partition tolerance isn't really optional in a real distributed system, so the actual choice in practice is CP vs AP.
    • Applying one CAP choice uniformly across an entire system, when different data within the same application often has genuinely different consistency requirements.
    • Confusing CAP's strict "consistency" (linearizability) with the everyday meaning of "the data is usually correct" — CAP consistency is a strong, specific guarantee, and most systems that claim to be "eventually consistent" are deliberately choosing AP, not failing at CP.
    Interview answer
    "Since network partitions are a real, unavoidable failure mode in any distributed system, the practical choice CAP forces is between consistency and availability specifically during that partition — not a free pick of any two. I don't apply that choice uniformly across a whole system though — for something like booking/inventory data, I'd lean CP, since briefly rejecting a request during a partition is a much smaller problem than two instances both confirming the same slot from stale data. For something like a view counter or an activity feed, AP is the right call — showing a slightly stale number is completely fine in exchange for staying available."
    Follow-up questions
    • Why isn't partition tolerance really "optional" in a real system?
    • Give an example of data in the same application that would reasonably be CP vs AP.
    • What does "eventually consistent" mean, and which side of CAP does it represent?
    CAP in practice during a network partition — pick one:

CP (Consistency + Partition tolerance):
  → reject/block the write until the partition heals
  → correct for: payments, inventory, booking confirmations

AP (Availability + Partition tolerance):
  → serve a response anyway, possibly stale, reconcile later
  → correct for: view counts, activity feeds, "last seen" indicators
    Best practices
    • Apply CAP trade-offs per data type based on real business impact of staleness vs unavailability, not uniformly across the whole system.
    • Be precise about what "consistency" means in context — CAP's definition is stronger than casual usage of the word.

    05. Database Replication & Sharding at Scale

    SENIOR

    Explanation. Replication keeps copies of the full dataset on multiple database nodes. Leader-follower (primary-replica) is most common: writes go to the leader, which propagates changes to followers; reads can be distributed across followers for read scaling, at the cost of replication lag — a follower might briefly serve stale data right after a write. Multi-leader replication accepts writes on multiple nodes (useful for multi-region setups) but introduces write-conflict resolution complexity. Sharding is fundamentally different — it splits the dataset itself horizontally across multiple database instances, each holding only a subset of the data, chosen by a shard key (e.g. user_id or a geographic region). Sharding is necessary when a single instance can't hold or serve the full dataset even with a powerful machine and read replicas; it's a much bigger architectural commitment since cross-shard joins/transactions become genuinely hard.

    Why interviewers ask it

    This tests whether a candidate reaches for sharding appropriately — it's frequently over-prescribed in interviews as a default "at scale" answer when replication (or better indexing/caching) solves the actual bottleneck far more simply, so knowing the real triggers for each separates genuine understanding from pattern-matched buzzwords.

    Real-world use case

    A booking platform's read-heavy search traffic is a textbook read-replica case — route search queries to followers, keep writes (actual bookings) on the leader — and this alone likely handles scale for a very long time before sharding is genuinely needed; sharding would only become necessary if the dataset itself grew too large for a single instance to hold, which is a fundamentally different problem than "too many reads."

    Common mistakes
    • Reaching for sharding as the default "how do you scale a database" answer when the actual bottleneck (read load) is much better solved by read replicas — a far smaller architectural change.
    • Not accounting for replication lag — reading your own just-written data from a follower immediately after the write can show stale results, a real bug class ("why doesn't my booking show up immediately") if not handled with read-after-write consistency where it matters.
    • Picking a shard key that creates hot spots — e.g. sharding by signup date when most active traffic is on recently-created accounts, concentrating load unevenly instead of distributing it.
    Interview answer
    "I reach for read replicas long before sharding — most real bottlenecks are read-heavy, and leader-follower replication solves that with a much smaller architectural footprint, distributing reads across followers while writes stay on the leader. The trade-off to be upfront about is replication lag — a follower can briefly serve stale data right after a write, so anywhere read-after-write consistency actually matters, like a user immediately seeing their own new booking, I'd route that specific read to the leader rather than a follower. Sharding I treat as a last resort, only once the dataset itself genuinely can't fit or be served by a single instance even with replicas — and if I do shard, the shard key choice is the whole game, since a bad key creates hot spots that just relocate the original problem."
    Follow-up questions
    • What's replication lag, and how would you handle a "read your own write" requirement despite it?
    • What makes a good vs bad shard key choice?
    • Why is sharding a bigger architectural commitment than adding read replicas?
    Leader-Follower Replication:

  Writes ──▶ [Leader] ──replicates──▶ [Follower 1]
                    └──replicates──▶ [Follower 2]
  Reads  ──▶ [Follower 1 or 2]   (scales read capacity, has lag risk)

Sharding by user_id (splits the DATASET, not just read load):

  user_id % 3 == 0 ──▶ [Shard A]
  user_id % 3 == 1 ──▶ [Shard B]
  user_id % 3 == 2 ──▶ [Shard C]
  (a query needing data across shards now needs a cross-shard fan-out)
    Best practices
    • Exhaust replication, indexing, and caching before considering sharding.
    • Route read-after-write-sensitive queries to the leader explicitly, rather than assuming follower reads are always safe.

    06. Caching Architecture & CDNs at Scale

    MID

    Explanation. At a system-design level, caching happens at multiple layers stacked together: CDN edge caching (static assets, sometimes full page responses, served from locations near the user), application-layer caching (Redis/Memcached in front of the database), and client-side/browser caching. A distributed cache like Redis can itself be scaled via replication and, at large scale, consistent hashing — a technique for distributing cache keys across multiple cache nodes such that adding or removing a node only reshuffles a small fraction of keys, instead of nearly all of them (which plain modulo-based hashing would cause, triggering a massive cache-miss storm). Eviction policies (LRU — evict least recently used, LFU — evict least frequently used, TTL-based) determine what gets removed when the cache is full.

    Why interviewers ask it

    At scale, caching stops being "add Redis" and becomes its own distributed system with its own failure modes — consistent hashing specifically is a strong signal of genuine distributed-systems depth versus surface-level "caching speeds things up" knowledge.

    Real-world use case

    A camp search results cache under heavy read traffic benefits from consistent hashing across multiple Redis nodes specifically because it means scaling the cache cluster up or down (adding a node during a traffic spike) doesn't invalidate almost the entire cache at once — with plain modulo hashing, adding one node to a 4-node cluster would remap roughly 80% of keys, causing a sudden flood of cache misses hitting the database simultaneously.

    Common mistakes
    • Using simple modulo-based hashing (hash(key) % numNodes) to distribute cache keys across nodes, causing almost total cache invalidation whenever a node is added or removed — exactly the problem consistent hashing solves.
    • Not planning for cache stampede — many requests simultaneously missing the same expired key and all hammering the database at once to recompute it; usually mitigated with a lock/single-flight pattern or staggered TTL jitter.
    • Caching at only one layer when a layered approach (CDN + application cache) would reduce load far more effectively at each stage.
    Interview answer
    "I think of caching as layered — CDN for static assets closest to the user, application-layer Redis in front of the database for computed/query results. Once the cache itself needs to scale across multiple nodes, the key detail is how keys get distributed — plain modulo hashing means adding or removing a node reshuffles almost every key at once, causing a massive simultaneous cache-miss storm against the database. Consistent hashing avoids that by only reassigning a small fraction of keys when the cluster changes size. I also plan explicitly for cache stampede — if a hot key expires and many requests miss it simultaneously, without a lock or staggered TTL, all of them hit the database at once for the same recomputation."
    Follow-up questions
    • Why does modulo-based hashing cause a cache-miss storm when a node is added?
    • What's a cache stampede, and how would you prevent one?
    • What's the difference between LRU and LFU eviction, and when would each be more appropriate?
    // Naive modulo hashing — adding a 4th node reshuffles ~75% of keys
nodeIndex = hash(key) % numNodes;

// Consistent hashing — nodes and keys both mapped onto a ring;
// adding/removing a node only affects keys in its adjacent ring segment,
// not the entire keyspace

// Cache stampede mitigation — single-flight lock around recomputation
async function getWithLock(key) {
  const cached = await redis.get(key);
  if (cached) return cached;

  const lockAcquired = await redis.set(`lock:${key}`, '1', 'NX', 'EX', 5);
  if (!lockAcquired) { await sleep(50); return getWithLock(key); } // wait, retry

  const fresh = await computeExpensiveValue();
  await redis.set(key, fresh, 'EX', 300);
  await redis.del(`lock:${key}`);
  return fresh;
}
    Best practices
    • Use consistent hashing for any cache cluster expected to scale up/down, to avoid mass cache invalidation.
    • Protect hot keys with a stampede-prevention mechanism (lock or jittered TTL) rather than assuming misses are always cheap.

    07. Message Queues, Pub/Sub & Event-Driven Architecture

    SENIOR

    Explanation. A message queue (RabbitMQ, SQS) delivers each message to exactly one consumer — good for distributing work across a pool of workers (a job queue). Pub/Sub (Kafka, Redis Pub/Sub, SNS) delivers each message to every subscriber independently — good for broadcasting an event to multiple independent systems that each need to react. Event-driven architecture builds a system around producing and reacting to events rather than direct service-to-service calls, decoupling producers from consumers entirely — a producer doesn't need to know who's listening or how many consumers exist. Kafka

    Why interviewers ask it

    Distinguishing queue-style (one consumer) from pub/sub-style (every subscriber) delivery is a common point of confusion, and getting it wrong in a system design leads to a genuinely broken architecture — like assuming every downstream service will receive an event when the actual technology only delivers it to one.

    Real-world use case

    A booking.created event needs to reach several independent systems — send a confirmation email, update analytics, notify the organizer, kick off the background check — which is a pub/sub/event-bus shape (every subscriber gets it), fundamentally different from processing a pool of image-resize jobs where each job should be picked up by exactly one worker, which is a traditional queue shape.

    Common mistakes
    • Using a plain message queue when multiple independent systems each need to react to the same event, not realizing a queue only delivers to one consumer — the second "subscriber" simply never receives messages the first one already consumed.
    • Not designing consumers to be idempotent, since most messaging systems guarantee at-least-once delivery, meaning duplicate processing is expected, not exceptional.
    • Choosing Kafka for a simple job-queue use case where its replay/ordering/durability guarantees add operational complexity with no real benefit over a simpler queue.
    Interview answer
    "The distinction I make immediately is whether this event needs to reach one consumer or every interested consumer — a job queue distributing image-processing work across a worker pool is one-consumer-per-message, so a traditional queue fits; a 'booking created' event that needs to trigger an email, update analytics, and notify an organizer independently is pub/sub, since every one of those needs its own copy of the event. Either way, I design consumers assuming at-least-once delivery and build in idempotency, since that's the standard guarantee most messaging systems actually make — exactly-once is the exception, not the default to assume."
    Follow-up questions
    • What actually breaks if you use a plain queue where you needed pub/sub?
    • What makes Kafka different from a traditional message queue like RabbitMQ?
    • Why does event-driven architecture make a system harder to reason about, even though it decouples services?
    Queue (one consumer per message) — worker pool pattern:

  [Producer] ──▶ [Queue: resize-jobs] ──▶ Worker 1  (picks up job A)
                                      ──▶ Worker 2  (picks up job B)
                  each message consumed by exactly ONE worker

Pub/Sub (every subscriber gets a copy) — event broadcast pattern:

  [booking.created event]
        │
        ├──▶ Email Service       (sends confirmation)
        ├──▶ Analytics Service   (logs the event)
        └──▶ Notification Service (alerts the organizer)
  EVERY subscriber independently receives its own copy
    Best practices
    • Choose queue vs pub/sub based explicitly on whether one or many consumers need each message — this is a hard architectural constraint, not a style preference.
    • Design every consumer assuming at-least-once delivery; idempotency is a requirement, not an optimization.

    08. Case Study: Design a URL Shortener

    MID

    Explanation. The core challenge is generating a short, unique, unpredictable-enough code and mapping it to a long URL with a lookup fast enough for real-time redirects. Two common approaches: base62 encoding of an auto-incrementing ID (simple, guaranteed unique, but sequential IDs are guessable/enumerable) or a hash of the URL truncated to a fixed length (needs collision handling, since truncated hashes can collide). The system is overwhelmingly read-heavy — far more redirects happen than new URLs get created — which directly shapes the design toward aggressive caching of the short-code → long-URL mapping and read-optimized storage.

    Why interviewers ask it

    It's a small, bounded problem that still touches ID generation, caching, read/write ratio reasoning, and database choice — making it a favorite for a 30-45 minute interview slot where there's time for a real deep dive, not just a surface sketch.

    Real-world use case

    Directly analogous to generating shareable booking confirmation links or referral codes on a platform like CampVine — the same core problem (short, unique, fast-lookup identifier mapped to a longer record) shows up constantly in real backend work beyond the specific "URL shortener" framing.

    Common mistakes
    • Not addressing collision handling at all when using a hash-based approach — truncated hashes can and will collide at scale, needing a fallback strategy.
    • Missing that the system is read-heavy and designing it as if reads and writes were roughly balanced — this should drive the caching strategy and possibly the choice to denormalize/cache aggressively.
    • Not considering custom aliases or expiration as functional requirements the interviewer might expect to be asked about.
    Interview answer
    "I'd clarify scale and whether custom aliases/expiration are needed, then go with base62-encoding an auto-incrementing ID for the short code — simple and collision-free by construction, unlike a truncated hash. Given this is overwhelmingly read-heavy — far more redirects than URL creations — I'd put a cache in front of the database for the short-code lookup, since that's the hot path on every single redirect, and the database itself just needs a straightforward indexed lookup table. For scale beyond a single instance, the ID generation itself needs to be distributed-safe — either a centralized ID generation service or pre-allocated ID ranges per server — so multiple app servers don't hand out the same ID."
    Follow-up questions
    • How would you generate unique IDs across multiple app server instances without collisions?
    • How would you support custom user-chosen aliases alongside auto-generated codes?
    • What happens to the cache when a URL is deleted or expires?
    High-level design:

  Client ──▶ [Load Balancer] ──▶ [API Servers] ──▶ [Cache: code → URL]
                                        │                    │ (miss)
                                        ▼                    ▼
                              [DB: id, short_code, long_url, created_at]

Short code generation:
  auto-increment ID (12345) ──▶ base62 encode ──▶ "3d7"
  (guaranteed unique, no collision handling needed, unlike hashing)

Read path (hot):  short_code ──▶ cache hit ──▶ 301/302 redirect
Write path (cold): long_url ──▶ generate ID ──▶ store ──▶ return short_code
    Best practices
    • Design the ID generation strategy to be safe across multiple distributed app servers from the start.
    • Cache the hot read path aggressively — this system's entire performance profile is dominated by redirect lookups.

    09. Case Study: Design a Chat Application

    SENIOR

    Explanation. A chat system needs real-time bidirectional delivery (WebSockets, as covered in Chapter 3), message persistence (so history survives a disconnect/reload), delivery guarantees (at minimum "delivered," ideally "read" receipts), and — the genuinely hard part at scale — routing a message to the right recipient regardless of which server instance holds their connection. Since WebSocket connections are stateful and pinned to a specific server instance, a message sent by User A has to be routed to whichever specific instance User B is currently connected to — solved with a pub/sub layer (Redis or Kafka) that every server instance subscribes to, broadcasting "deliver this message to user B" so the instance actually holding B's connection can pick it up and push it.

    Why interviewers ask it

    This case study specifically tests whether a candidate understands the operational reality of stateful connections at scale — it's a natural continuation of the WebSocket horizontal-scaling problem, and a strong candidate connects those dots without being prompted.

    Real-world use case

    Directly relevant to a real feature like organizer-to-guest messaging on a booking platform — the exact same architecture (WebSocket + pub/sub cross-instance routing + persisted message history) would be needed the moment that feature needed to run on more than one server instance.

    Common mistakes
    • Designing the WebSocket layer without addressing how a message reaches a recipient connected to a different server instance than the sender — the single hardest and most commonly glossed-over part of this problem.
    • Not separating the real-time delivery path from persistence — a message should be durably stored (so it survives a server restart or appears correctly on reload) independent of whether the real-time push succeeds.
    • Ignoring offline delivery — what happens to a message sent to a currently-disconnected user; it needs to be queryable/delivered on their next reconnect, not silently lost.
    Interview answer
    "The real-time transport is WebSockets, but the part that actually makes this hard at scale is that those connections are stateful and pinned to a specific server instance — so when User A sends a message, the server handling A's connection doesn't necessarily hold B's connection too. I'd solve that with a pub/sub layer like Redis that every instance subscribes to: A's server publishes the message, and whichever instance actually holds B's connection picks it up and pushes it over that socket. Separately, and independently of whether that real-time push succeeds, the message gets durably written to a database — so chat history survives reconnects and a message to an offline user is queryable and deliverable the next time they connect, rather than being lost if the push simply had no live socket to reach."

    10. Case Study: Design a Notification System & Rate Limiter

    SENIOR

    Explanation. A notification system fans a single triggering event out across multiple channels (push, email, SMS, in-app) with very different delivery characteristics — different latency, different failure modes, different providers — which is why it's naturally an event-driven, queue-backed design: the triggering event goes onto a queue/bus, and independent workers per channel consume it and handle their own provider-specific delivery and retry logic without blocking each other. A rate limiter is often a required sub-component (don't spam a user with duplicate notifications, respect a downstream provider's own rate limits) — commonly implemented with the token bucket algorithm (a bucket refills at a fixed rate and each request consumes a token; requests are rejected once the bucket is empty, but bursts up to the bucket size are allowed) or a sliding window counter (more precise, more computationally expensive) tracked in Redis for shared state across server instances.

    Why interviewers ask it

    This combines several earlier topics (event-driven architecture, queues, idempotency) into one cohesive design, and the rate limiter sub-problem specifically is a very common standalone interview question in its own right, testing precise algorithmic thinking under a real constraint.

    Real-world use case

    A booking confirmation triggers exactly this fan-out — email confirmation, push notification to the organizer, in-app notification badge — each handled by an independent consumer so a slow/down email provider doesn't delay or block the push notification from going out; and a per-user rate limit on notifications prevents a bug or retry storm from spamming a single user with the same alert dozens of times.

    Common mistakes
    • Sending to all channels synchronously and sequentially in one request handler, meaning a slow or down provider for one channel (say, SMS) delays or blocks the others entirely.
    • Not deduplicating/rate-limiting notifications, so a retry or a bug causes the same alert to be sent to a user many times — a real, user-visible reliability problem, not just an edge case.
    • Implementing a rate limiter with local in-memory state on a single server instance, which doesn't work correctly the moment there's more than one instance — needs shared state (Redis) so the limit is enforced consistently across all of them.
    Interview answer
    "I'd design this as event-driven from the trigger outward — a booking-confirmed event goes onto a bus, and independent consumers per channel (email, push, SMS) each handle their own delivery and retry logic, so a slow or failing provider on one channel never blocks the others. For the rate limiting piece specifically, I'd use a token-bucket algorithm backed by Redis rather than in-memory state, since the limit needs to be enforced consistently across however many server instances are running — in-memory counters would let each instance independently think the user is under the limit, effectively multiplying the real limit by the number of instances. Token bucket specifically because it allows reasonable bursts while still capping the sustained rate, which fits notification behavior better than a hard fixed-window cutoff."
    Follow-up questions
    • Why would a rate limiter using local in-memory state break across multiple server instances?
    • Walk through exactly how the token bucket algorithm works.
    • How would you prevent a user from receiving the same notification twice due to a retry?
    // Token bucket rate limiter, Redis-backed for shared state across instances
async function allowRequest(userId, bucketSize = 10, refillPerSec = 1) {
  const key = `ratelimit:${userId}`;
  const now = Date.now() / 1000;

  const bucket = await redis.hgetall(key) || { tokens: bucketSize, lastRefill: now };
  const elapsed = now - bucket.lastRefill;
  const tokens = Math.min(bucketSize, bucket.tokens + elapsed * refillPerSec);

  if (tokens < 1) return false; // rate limited — bucket empty

  await redis.hset(key, { tokens: tokens - 1, lastRefill: now });
  return true; // allowed, one token consumed
}

// Notification fan-out — independent consumers per channel
eventBus.publish('booking.confirmed', { bookingId, userId });
// EmailWorker, PushWorker, and SMSWorker each subscribe independently —
// a slow SMS provider never blocks email or push delivery
    Best practices
    • Fan out to delivery channels independently via a queue/event bus, never synchronously in one blocking request.
    • Back any rate limiter with shared state (Redis), never local in-memory counters, once more than one server instance is in play.
