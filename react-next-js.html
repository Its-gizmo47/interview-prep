    01. Virtual DOM, Reconciliation & JSX

    MID

    Explanation. JSX is syntax sugar that compiles to React.createElement() calls, producing plain JS objects that describe UI — not actual DOM nodes. The Virtual DOM is React's lightweight in-memory representation of that tree. On every state change, React builds a new virtual tree and runs reconciliation — a diffing algorithm comparing the new tree to the previous one — to compute the minimal set of real DOM mutations needed, then applies just those, rather than re-rendering the entire page. Reconciliation relies on heuristics for speed: elements of a different type at the same position are assumed entirely different (old subtree torn down, new one built), and lists rely on the key prop to match elements across renders instead of comparing by position alone.

    Why interviewers ask it

    This is the conceptual foundation for almost every React performance question that follows — re-renders, keys, memoization all only make sense once you understand what reconciliation is actually doing and why it needs help identifying what changed.

    Real-world use case

    Understanding that changing an element's type (say, swapping a <div> wrapper for a <section> conditionally) causes React to tear down and rebuild the entire subtree — including losing all component state and re-running effects inside it — explains a real, easy-to-miss bug class where conditionally swapping wrapper elements silently resets form input state.

    Common mistakes
    • Believing the Virtual DOM makes React "fast" in some absolute sense — it doesn't skip work by magic, it minimizes and batches real DOM mutations, which are the genuinely expensive operation.
    • Not realizing that changing a component's position in the tree (or its type) can unmount and remount it entirely, resetting local state and re-triggering effects.
    • Thinking JSX is HTML — it's syntax that compiles to function calls, which is why expressions need {} and why certain HTML attributes (like class → className) differ.
    Interview answer
    "JSX compiles down to React.createElement calls that build a lightweight object tree — the virtual DOM — rather than touching the real DOM directly. When state changes, React builds a new virtual tree and diffs it against the previous one during reconciliation, computing the minimal real DOM changes needed and batching them. The part people miss is that this diffing relies on heuristics — same position, same type is assumed to be the same logical element and gets updated in place; a different type at that position tears the whole subtree down and rebuilds it, which is why conditionally swapping a wrapper element can silently reset child state."
    Follow-up questions
    • What happens to a component's local state if its type changes at the same tree position?
    • Why does React need a diffing algorithm at all instead of just re-rendering everything to the real DOM?
    • Is JSX required to use React? What does it compile to?
    // JSX...
const element = <h1 className="title">Hello, {name}</h1>;

// ...compiles to this (roughly)
const element = React.createElement('h1', { className: 'title' }, 'Hello, ', name);

// Type change at the same position tears down + rebuilds the subtree,
// losing all state and effects inside it — a real, surprising bug source
{ isEditing ? <input value={text} /> : <div>{text}</div> }
    Best practices
    • Keep the same element type across conditional renders when you want to preserve internal state — vary content, not structure, where possible.
    • Remember the virtual DOM's job is minimizing real DOM writes — it doesn't make expensive render logic itself free; that's what memoization solves separately.

    02. Props, State & Controlled vs Uncontrolled Components

    MID

    Explanation. Props are read-only data passed down from a parent — a component must never mutate its own props. State is data owned and managed internally by a component (via useState/useReducer) that can change over time and triggers a re-render when it does. A controlled component has its value driven entirely by React state — the input's value comes from state and every change goes through an onChange handler that updates that state, making React the single source of truth. An uncontrolled component lets the DOM itself hold the current value, accessed on demand via a ref instead of tracked on every keystroke — simpler for a case like a one-time file upload or a form you only read on submit.

    Why interviewers ask it

    Forms are one of the most common real UI patterns, and this distinction directly determines architecture — get it wrong and you either fight React's rendering model (uncontrolled input trying to sync to state) or add unnecessary re-renders and complexity (controlling every field when you only needed the value on submit).

    Real-world use case

    A multi-step booking form with live validation (checking availability as the user types dates) needs controlled inputs so every keystroke is available to validate and drive UI feedback immediately, while a simple file upload field in that same form is a natural fit for an uncontrolled ref-based input, since you typically only need its value at submit time, not on every change.

    Common mistakes
    • Directly mutating a prop or an object/array received as a prop, which can cause bugs since the parent's state changes underneath it without a proper re-render being triggered through React's data flow.
    • Mixing controlled and uncontrolled on the same input — setting a value without an onChange handler makes React log a warning and effectively locks the field (or throws it into an inconsistent state).
    • Making every field controlled by default even when the value is only needed once, adding unnecessary re-renders on every keystroke for no real benefit.
    Interview answer
    "Props flow down and are read-only — a component should never mutate them, only its parent controls that data. State is owned locally and its changes are what actually drive re-renders. For forms specifically, I go controlled when I need the current value on every change — live validation, conditional fields, character counts — and uncontrolled with a ref when I genuinely only need the value once, like on submit, since that avoids a re-render per keystroke for data nobody's reading in between."
    Follow-up questions
    • What warning does React give if you set a value prop on an input with no onChange handler, and why?
    • Can a component modify a prop it receives? What happens if you try?
    • When would you deliberately choose an uncontrolled input over a controlled one?
    // Controlled — React state is the single source of truth
function ControlledInput() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// Uncontrolled — DOM holds the value, read via ref only when needed
function UncontrolledInput() {
  const inputRef = useRef(null);
  const handleSubmit = () => console.log(inputRef.current.value);
  return <input ref={inputRef} defaultValue="" />;
}
    Best practices
    • Never mutate props — always work with a local copy or lift the change up via a callback prop.
    • Reach for uncontrolled + refs for simple, submit-only fields; controlled for anything needing live reaction to every change.

    03. Core Hooks: useState, useEffect, useRef, useReducer

    MID

    Explanation. useState holds a piece of local state and a setter that schedules a re-render; state updates within the same event handler are batched and setter calls using the previous value should use the functional form (setCount(c => c + 1)) to avoid stale-closure bugs. useEffect runs side effects (data fetching, subscriptions, manual DOM work) after render, re-running whenever its dependency array changes — an empty array means "once, on mount," no array means "every render," and the returned function is cleanup, run before the next effect and on unmount. useRef holds a mutable value that persists across renders without triggering a re-render when it changes — used for DOM references or any "instance variable" that shouldn't cause a render. useReducer is useState's more structured sibling for complex state logic, following the reducer pattern (state + action → new state), useful when multiple sub-values update together or the next state depends heavily on the action type.

    Why interviewers ask it

    Hooks are the daily bread-and-butter of modern React, and the dependency array specifically is the single most common source of real, subtle React bugs — stale closures, infinite loops, missing cleanup — so fluency here is non-negotiable for any React role.

    Real-world use case

    Fetching booking availability when a selected date changes is a useEffect with [selectedDate] as its dependency, with a cleanup function that ignores a stale in-flight response if the date changes again before the first request resolves — a real race-condition bug that shows up constantly in date-picker-driven UIs without that cleanup.

    Common mistakes
    • Omitting a value used inside useEffect from its dependency array — causes the effect to run with a stale closure over an old value, a very common real bug (and what the exhaustive-deps ESLint rule exists to catch).
    • Not cleaning up subscriptions/async requests in useEffect's cleanup function, causing state updates on an unmounted component or race conditions between overlapping requests.
    • Using useState's previous value directly (setCount(count + 1)) instead of the functional form inside rapid or batched updates, leading to lost updates.
    • Reaching for useRef to store a value and expecting the UI to update when it changes — refs don't trigger re-renders by design.
    Interview answer
    "The dependency array is where most real useEffect bugs live — I treat the exhaustive-deps lint rule as a signal, not noise, because omitting a value almost always means the effect is closing over a stale version of it. For anything async inside an effect — a fetch tied to a changing prop like a selected date — I always return a cleanup function that can ignore a stale response, otherwise a fast-changing input can cause an old request to resolve after a newer one and overwrite the correct state. I use useReducer over useState once I have several related pieces of state that update together based on distinct actions — it keeps that logic in one place instead of scattered across multiple setState calls."
    Follow-up questions
    • What's a stale closure in the context of useEffect, and how do you avoid it?
    • Why would you use the functional form of a useState setter instead of referencing state directly?
    • When would you reach for useReducer instead of several useState calls?
    useEffect(() => {
  let cancelled = false;
  fetchAvailability(selectedDate).then(data => {
    if (!cancelled) setAvailability(data); // ignore stale response if date changed again
  });
  return () => { cancelled = true; }; // cleanup — runs before next effect / on unmount
}, [selectedDate]);

// Functional setState form avoids stale-closure bugs in rapid updates
setCount(prevCount => prevCount + 1);

// useReducer for related state that updates together via distinct actions
function bookingReducer(state, action) {
  switch (action.type) {
    case 'SET_DATE': return { ...state, date: action.payload, error: null };
    case 'SET_ERROR': return { ...state, error: action.payload };
    default: return state;
  }
}
    Best practices
    • Trust the exhaustive-deps ESLint rule; if you're tempted to disable it, that's usually a sign the effect should be restructured, not a sign the rule is wrong.
    • Always clean up subscriptions/timers/in-flight-request tracking in useEffect's return function.

    04. useMemo, useCallback & Re-render Optimization

    SENIOR

    Explanation. A component re-renders when its own state changes, its parent re-renders (by default — children re-render whenever the parent does, regardless of whether their own props changed), or its context value changes. useMemo caches the result of an expensive computation between renders, only recomputing when its dependencies change. useCallback caches a function reference itself, which matters because a new function is created on every render by default — passing a fresh function reference as a prop to a memoized child defeats that memoization, since React.memo's shallow comparison sees a "new" prop every time even if the function's logic is identical. React.memo wraps a component so it skips re-rendering if its props are shallowly equal to the previous render.

    Why interviewers ask it

    Misusing (or under-using) these hooks is extremely common — either wrapping everything in useMemo/useCallback "for performance" with no measured need, adding complexity and its own overhead for nothing, or missing them where they genuinely matter and shipping a visibly janky UI on a large list or expensive computation.

    Real-world use case

    A camp search results page filtering/sorting a large in-memory list on every keystroke of a search box is a legitimate useMemo case — recomputing the filtered+sorted list only when the search term or source data actually changes, not on every unrelated re-render of the parent (like a sidebar count updating).

    Common mistakes
    • Wrapping cheap computations or trivial functions in useMemo/useCallback "just in case" — the memoization machinery itself has a cost, and for cheap operations it's a net loss, not a win.
    • Memoizing a component with React.memo but still passing it a new object/array/function literal as a prop on every render, silently defeating the memoization since shallow comparison sees a new reference each time.
    • Treating memoization as a default habit instead of a response to a measured, real performance problem — usually found via the React DevTools Profiler, not guessed at.
    Interview answer
    "I don't reach for useMemo or useCallback by default — I profile first with React DevTools and only optimize a re-render that's actually shown up as a real problem, usually an expensive computation over a large list or a component that's visibly janky. When I do use them, the thing I watch for is that they only pay off together — memoizing a child with React.memo does nothing if the parent keeps passing it a brand-new function or object reference every render, so useCallback and useMemo on the parent side are what actually make that memoization stick."
    Follow-up questions
    • Why would wrapping a child in React.memo sometimes have zero effect on preventing re-renders?
    • When is useMemo actually a performance loss rather than a gain?
    • How would you find out whether a component's re-renders are actually a real performance problem?
    // useMemo — only recompute the filtered list when term or camps actually change
const filteredCamps = useMemo(
  () => camps.filter(c => c.name.includes(searchTerm)),
  [camps, searchTerm]
);

// Without useCallback, this new function reference on every render
// defeats React.memo on , causing it to re-render anyway
const handleSelect = useCallback(
  (id) => setSelectedId(id),
  [] // stable reference across renders
);

const CampCard = React.memo(function CampCard({ camp, onSelect }) {
  return <div onClick={() => onSelect(camp.id)}>{camp.name}</div>;
});
    Best practices
    • Profile before optimizing — treat useMemo/useCallback as a targeted fix, not a default habit.
    • When memoizing a child component, memoize the props you pass it too — object/array/function literals need useMemo/useCallback on the parent side to actually stay stable.

    05. Custom Hooks & Context API

    MID

    Explanation. A custom hook is just a regular function whose name starts with use that calls other hooks internally — it's React's mechanism for extracting and reusing stateful logic across components without changing the component tree (unlike the older render-props or higher-order-component patterns). The Context API solves prop drilling — passing a value through many intermediate components that don't need it — by letting a Provider make a value available to any descendant that calls useContext, no matter how deep. The trade-off: any component consuming a context re-renders whenever that context's value changes, even if the specific field it reads didn't change — which is why Context isn't a full replacement for a proper state management solution in a large, frequently-updating app.

    Why interviewers ask it

    Extracting logic into custom hooks is the primary way experienced React developers keep components clean and testable, and understanding Context's re-render trade-off signals whether you can make an informed architectural call between Context, prop drilling, and a dedicated state library rather than reaching for Context by default.

    Real-world use case

    A useDebounce or useAvailability(campId, date) custom hook wrapping the fetch/loading/error logic for checking camp availability is directly reusable across a search page, a booking form, and a calendar view — without copy-pasting the same useEffect/useState logic three times.

    Common mistakes
    • Putting a frequently-changing value (like real-time form input) in Context, causing every consumer anywhere in the tree to re-render on every keystroke.
    • Reaching for Context as a general-purpose global state manager for everything, when a large, frequently-updating piece of state is usually better served by a dedicated library (or splitting into multiple smaller, more targeted contexts).
    • Calling hooks conditionally or inside loops — breaks React's reliance on stable hook call order between renders (the Rules of Hooks).
    Interview answer
    "I reach for a custom hook the moment I see the same stateful logic — a fetch-with-loading-and-error pattern, a subscription, a debounce — duplicated across two or more components; it's just a function that calls other hooks, so it composes cleanly without adding to the component tree the way older patterns like HOCs did. For Context, I'm deliberate about what goes in it, because every consumer re-renders on any change to the context value — so I keep frequently-changing state out of Context, or split large context objects into smaller, more targeted ones so an update to one slice doesn't re-render every consumer of the whole thing."
    Follow-up questions
    • Write a custom hook that debounces a value.
    • Why does every Context consumer re-render on any change to the context value, even unrelated fields?
    • What are the Rules of Hooks, and why do they exist?
    // Custom hook — reusable fetch-with-status logic
function useAvailability(campId, date) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    fetchAvailability(campId, date).then(result => {
      if (!cancelled) { setData(result); setLoading(false); }
    });
    return () => { cancelled = true; };
  }, [campId, date]);

  return { data, loading };
}

// Usage — identical logic, reused across 3 different components
const { data, loading } = useAvailability(campId, selectedDate);
    Best practices
    • Extract duplicated stateful logic into a custom hook as soon as it appears in a second component.
    • Split large context objects into smaller, purpose-specific contexts to limit unnecessary re-renders.

    06. Keys, Lists & Component Rendering Behavior

    MID

    Explanation. When rendering a list, the key prop tells React which array item corresponds to which rendered element across renders, so it can correctly match, reorder, insert, or remove elements instead of guessing by position. Using the array index as a key works fine for a static list that never reorders, but breaks down the moment items can be added, removed, or reordered — because the index-to-item mapping shifts, React can misattribute internal state (like an input's typed value) to the wrong logical item after a reorder. A stable, unique identifier (a database ID, not the array index) keeps that mapping correct regardless of how the list changes.

    Why interviewers ask it

    The index-as-key bug is one of the most common, subtle React bugs in real codebases — it doesn't throw an error, it just silently produces wrong behavior (state attached to the wrong row) that's easy to miss in casual testing and only shows up once a user actually reorders or deletes something.

    Real-world use case

    A list of editable booking line items where each row has its own local input state (like a quantity field) — using index as key means deleting the second row shifts every subsequent row's index down by one, and React reuses each row's existing DOM/state at that shifted index, silently showing the wrong quantity value on the wrong camp/item.

    Common mistakes
    • Using array index as key on any list that can reorder, filter, or have items inserted/removed — the single most common key-related bug.
    • Using a non-unique or non-stable value as a key (like the item's display name, which could repeat or change) instead of a genuinely stable unique ID.
    • Generating a new key on every render (like Math.random() or a fresh object per render) — defeats the entire purpose of keys, since nothing is stable across renders anymore, forcing React to tear down and rebuild every item every time.
    Interview answer
    "Keys are how React matches array items across renders instead of assuming position means identity. Index-as-key works fine for a genuinely static list, but the moment items can be reordered, inserted, or removed, the index-to-item mapping shifts and React can attach the wrong internal state — like a typed input value — to what's now a different logical item at that index. I always use a stable, unique identifier from the actual data, like a database ID, specifically so reordering or deleting an item doesn't corrupt state on the rows that didn't move."
    Follow-up questions
    • Walk through exactly what goes wrong if you delete the 2nd item from a 5-item list keyed by index, where each item has its own input state.
    • Is it ever actually fine to use index as key? When?
    • What happens if you don't provide a key prop at all on a list?
    // Risky: index as key on a list that can reorder/delete
{items.map((item, index) => <BookingRow key={index} item={item} />)}

// Correct: stable unique ID from the data itself
{items.map(item => <BookingRow key={item.id} item={item} />)}

// Index-as-key is fine ONLY when the list is guaranteed static —
// no reordering, no insertion/removal, no filtering, ever
{['Mon', 'Tue', 'Wed'].map((day, i) => <span key={i}>{day}</span>)}
    Best practices
    • Default to a stable unique ID from the data as the key; only use index for genuinely static, never-reordered lists.
    • Never generate a new key value on every render — that defeats the purpose entirely.

    07. Error Boundaries, Suspense & Lazy Loading

    SENIOR

    Explanation. An Error Boundary is a class component (no hook equivalent exists as of stable React) implementing componentDidCatch/getDerivedStateFromError that catches JavaScript errors thrown anywhere in its child tree during rendering, preventing one broken component from crashing the entire app — it renders a fallback UI instead. It does not catch errors in event handlers, async code, or server-side rendering — those need regular try/catch. React.lazy() + dynamic import() code-splits a component so its bundle is only fetched when actually needed, paired with Suspense, which shows a fallback UI while the lazy component (or, in newer React/Next.js, any async data-fetching boundary) is loading.

    Why interviewers ask it

    Production apps need graceful degradation — one buggy widget shouldn't take down an entire page — and code-splitting is a direct, measurable lever on initial load performance, so both are practical, "have you shipped a real app" signals rather than academic ones.

    Real-world use case

    Wrapping a third-party embed (like a map or a payment widget) in its own Error Boundary means a failure in that specific integration shows a contained fallback instead of white-screening the entire booking page — genuinely important when depending on external, less-controllable code. Lazy-loading a heavy admin dashboard route that most users never visit keeps it out of the main bundle entirely.

    Common mistakes
    • Expecting an Error Boundary to catch an error thrown inside an onClick handler or inside a .then() — it only catches errors during rendering/lifecycle methods/constructors of its children, not async or event-handler code.
    • Putting a single Error Boundary around the entire app with no finer granularity, so any error anywhere blanks the whole page instead of isolating the failure to the actual broken widget.
    • Lazy-loading components that are needed immediately on initial render, adding an unnecessary loading flash for no bundle-size benefit.
    Interview answer
    "I place Error Boundaries around specific, isolated units — especially anything depending on a third-party integration or a feature that's genuinely optional to the rest of the page — rather than one boundary around the whole app, so a failure stays contained to that widget instead of blanking everything. The scope people get wrong is thinking it catches everything; it only catches errors during rendering in its child tree, not inside event handlers or promises, which still need their own try/catch. For lazy loading, I only split out things that aren't needed on initial render — a heavy route most users don't visit, a modal's content — since splitting something needed immediately just adds a loading flash with no real benefit."
    Follow-up questions
    • Does an Error Boundary catch an error thrown inside a button's onClick handler? Why or why not?
    • Why is there no hook equivalent for Error Boundaries?
    • What's the difference between Suspense for lazy-loaded components and Suspense for data fetching?
    class ErrorBoundary extends React.Component {
  state = { hasError: false };
  static getDerivedStateFromError() { return { hasError: true }; }
  componentDidCatch(error, info) { logToService(error, info); }
  render() {
    if (this.state.hasError) return <p>Something went wrong loading this section.</p>;
    return this.props.children;
  }
}

// Isolate a risky third-party widget instead of one app-wide boundary
<ErrorBoundary>
  <PaymentWidget />
</ErrorBoundary>

// Code-splitting a rarely-used route
const AdminDashboard = React.lazy(() => import('./AdminDashboard'));

<Suspense fallback={<Spinner />}>
  <AdminDashboard />
</Suspense>
    Best practices
    • Scope Error Boundaries to isolated, meaningful sections — especially third-party or genuinely optional widgets — not just one boundary for the whole app.
    • Only lazy-load what isn't needed on initial render; measure with a bundle analyzer rather than splitting everything by default.

    08. Next.js Rendering Strategies: SSR, SSG, ISR, CSR

    SENIOR

    Explanation. CSR (Client-Side Rendering) ships a near-empty HTML shell and lets the browser fetch JS and render everything — fast to serve, slow to first meaningful paint, poor for SEO without extra work. SSR (Server-Side Rendering) generates the full HTML on the server for every request — good SEO and fast first paint, but adds server load and latency per request since it's redone every time. SSG (Static Site Generation) generates the HTML once at build time — extremely fast to serve (often from a CDN) and cheap on infrastructure, but the content is frozen until the next build/deploy. ISR (Incremental Static Regeneration) is the middle ground: pages are statically generated but automatically regenerated in the background after a configured time interval (or on-demand), so you get static-speed serving with content that doesn't go permanently stale.

    Why interviewers ask it

    Choosing the right rendering strategy per page/route is one of the most consequential real architectural decisions in a Next.js app, directly trading off SEO, performance, infrastructure cost, and content freshness — and picking wrong (SSR everywhere, or SSG for genuinely dynamic content) is a common, costly mistake.

    Real-world use case

    A camp listing/detail page is a strong ISR candidate — content changes occasionally (new bookings affect availability, a description gets edited) but not on every single request, so serving a statically-cached page that regenerates every 60 seconds gives near-static speed without showing hours-old, wrong availability data. A user's personal booking dashboard, by contrast, is genuinely per-user and needs SSR (or client-side fetching) since it can't be shared across visitors as a static page at all.

    Common mistakes
    • Using SSR by default for every page "to be safe," adding unnecessary server load and latency to pages that are actually the same for every visitor and could be statically generated instead.
    • Using SSG for content that changes frequently or is per-user, serving stale or completely wrong data until the next build.
    • Not understanding that ISR's staleness window means the very first request after the revalidation period can still briefly serve the old cached version while regeneration happens in the background (stale-while-revalidate behavior) — not an instant global update.
    Interview answer
    "I pick the rendering strategy per route based on how often the content changes and whether it's shared across users. Fully static marketing pages get SSG. Something like a camp listing page — public, cacheable, but not frozen forever — is a strong ISR candidate, since it serves at static speed but regenerates on an interval so availability data doesn't go stale for hours. Genuinely per-user content, like a personal booking dashboard, can't be a shared static page at all, so that's SSR or client-side fetched. The mistake I actively avoid is defaulting to SSR everywhere just because it feels 'safest' — it adds real per-request server cost and latency that most pages don't actually need."
    Follow-up questions
    • Which rendering strategy would you pick for a public blog vs a user's private dashboard vs a product listing page, and why?
    • What actually happens on the first request after an ISR page's revalidation window expires?
    • What are the SEO implications of pure CSR, and how would you mitigate them?
    // App Router: SSG by default for a route with no dynamic data functions
export default function CampsPage() { /* statically generated at build time */ }

// ISR: revalidate this route's cached data at most every 60 seconds
export const revalidate = 60;

async function getCamp(id) {
  const res = await fetch(`https://api.example.com/camps/${id}`, {
    next: { revalidate: 60 } // per-fetch ISR configuration
  });
  return res.json();
}

// SSR: force fresh data on every single request — no caching
async function getUserBookings(userId) {
  const res = await fetch(`https://api.example.com/users/${userId}/bookings`, {
    cache: 'no-store'
  });
  return res.json();
}
    Best practices
    • Choose per-route based on actual data volatility and whether content is shared or per-user — not a single blanket strategy for the whole app.
    • Use ISR for public, cacheable, occasionally-changing content as the default middle ground before reaching for full SSR.

    09. App Router: Server vs Client Components, Data Fetching & Caching

    SENIOR

    Explanation. In the App Router, every component is a Server Component by default — it renders on the server, ships zero JS to the client for that component, and can directly await data (database calls, file system, internal APIs) without a separate API route or client-side fetch. Adding 'use client' at the top of a file opts that component (and everything it imports) into being a Client Component — needed for interactivity (onClick, state, effects, browser-only APIs). The practical rule: keep components server-rendered by default, and push 'use client' as far down the tree as possible, wrapping only the actually-interactive leaf components, so the bulk of the page ships no JS. Next.js extends the native fetch with its own caching layer — { cache: 'force-cache' } (default, roughly SSG-like), { cache: 'no-store' } (SSR-like, always fresh), and { next: { revalidate: N } } (ISR-like) — letting caching behavior be configured per individual fetch call rather than per whole page.

    Why interviewers ask it

    This is the single biggest mental model shift from the older Pages Router, and misapplying 'use client' too broadly (a common instinct from pre-App-Router habits) silently defeats a large part of the App Router's actual performance benefit — so it's a strong, current signal of whether someone has real hands-on App Router experience versus older Next.js knowledge only.

    Real-world use case

    A camp detail page can be almost entirely a Server Component directly awaiting the camp data — only the "Book Now" button and date picker actually need interactivity, so those get isolated into a small Client Component while the surrounding description, images, and reviews stay server-rendered and ship no JS at all for that content.

    Common mistakes
    • Adding 'use client' at the top of a large page component just because one small piece of it needs interactivity, dragging the entire subtree into client-side JS unnecessarily instead of isolating just the interactive leaf.
    • Trying to use hooks like useState/useEffect or browser APIs inside a Server Component — they simply aren't available there, since it never runs in the browser.
    • Not understanding that Next.js's extended fetch caching is opt-out (cached by default), which surprises developers expecting always-fresh data by default, the opposite of typical client-side fetch behavior.
    Interview answer
    "My default is a Server Component for everything, and I only reach for 'use client' on the smallest possible leaf that actually needs interactivity — state, event handlers, browser APIs — rather than marking an entire page client-side because one button needs an onClick. That keeps most of the page's JS payload at zero, since Server Components render to HTML on the server and never ship their code to the browser at all. For data fetching, I'm deliberate about each fetch's cache option individually rather than assuming one caching behavior for the whole page — Next.js caches fetch calls by default, which catches people off guard if they're expecting client-fetch-style always-fresh behavior."
    Follow-up questions
    • Can a Server Component import and render a Client Component? Can it go the other way?
    • Why can't you use useState inside a Server Component?
    • What's the default caching behavior of fetch in the App Router, and how do you opt out of it?
    // Server Component (default, no directive needed) — awaits data directly
async function CampPage({ params }) {
  const camp = await getCamp(params.id); // direct DB/API call, zero client JS for this
  return (
    <div>
      <h1>{camp.name}</h1>
      <p>{camp.description}</p>
      <BookingButton campId={camp.id} /> {/* only THIS is client-side */}
    </div>
  );
}

// Isolated Client Component — the only piece that needs interactivity

  const [loading, setLoading] = useState(false);
  return <button onClick={() => setLoading(true)}>Book Now</button>;

    Best practices
    • Push 'use client' as far down the tree as possible — isolate interactivity in small leaf components.
    • Set each fetch call's cache behavior deliberately rather than assuming a default across the whole page.

    10. Next.js Middleware, Auth & Metadata

    MID

    Explanation. Next.js Middleware runs before a request completes, at the edge, before it hits a page or API route — used for auth redirects, A/B test bucketing, geolocation-based routing, or header manipulation, without needing to duplicate that logic inside every individual route. Auth in Next.js commonly combines middleware (checking for a valid session/token cookie and redirecting unauthenticated users before any page code even runs) with server-side session validation inside Server Components/Route Handlers for the actual data-fetching authorization. The Metadata API lets you define per-page <title>, description, and Open Graph tags declaratively — either statically via an exported metadata object, or dynamically via a generateMetadata function when the values depend on fetched data (like a camp's actual name and image for its detail page's social preview).

    Why interviewers ask it

    These are the practical, ship-it details that separate "built a React app" from "shipped a production Next.js app with real SEO and auth requirements" — metadata specifically has direct business impact (social sharing previews, search ranking) that's easy to get wrong or forget entirely.

    Real-world use case

    Protecting a camp organizer's dashboard routes with middleware that checks for a valid session cookie and redirects to /login before any dashboard code runs is both a better user experience (instant redirect, no flash of protected content) and more efficient than letting the page render and then checking auth client-side. Dynamic metadata on a camp detail page means sharing that link on social media shows the actual camp's name and photo instead of a generic site-wide title.

    Common mistakes
    • Relying only on client-side auth checks (redirecting after the page has already rendered) instead of middleware — causes a visible flash of protected content before the redirect happens, and is trivially bypassable since it's a UI-only check.
    • Treating a middleware-level cookie check as sufficient authorization for actual data access — middleware is a fast first gate, but sensitive server-side data fetching still needs its own proper session/permission validation, since middleware can be a shallow check by design (for performance at the edge).
    • Forgetting dynamic metadata entirely on data-driven pages, leaving every shared link showing the same generic title/image regardless of the actual content.
    Interview answer
    "For auth I use middleware as the first, fast gate — checking for a valid session cookie and redirecting before any protected page code even runs, which avoids the flash-of-content problem a purely client-side check has. But I don't treat that as the only check — the actual data-fetching in Server Components or Route Handlers still validates the session and permissions properly, since middleware is meant to be a lightweight edge check, not the full authorization logic. For metadata, anything data-driven — a camp detail page, a blog post — gets generateMetadata pulling the real title, description, and image from the fetched data, so shared links actually reflect the specific content instead of a generic fallback."
    Follow-up questions
    • Why isn't a middleware-level auth check alone sufficient for protecting sensitive data access?
    • What's the difference between the static metadata export and generateMetadata, and when do you need the latter?
    • What problem does middleware-based auth solve that a client-side redirect check doesn't?
    // middleware.ts — runs before the request reaches the page
export function middleware(request) {
  const session = request.cookies.get('session');
  if (!session && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
}
export const config = { matcher: ['/dashboard/:path*'] };

// Dynamic metadata pulling real content for social sharing
export async function generateMetadata({ params }) {
  const camp = await getCamp(params.id);
  return {
    title: camp.name,
    description: camp.description,
    openGraph: { images: [camp.imageUrl] },
  };
}
    Best practices
    • Use middleware for fast, edge-level auth redirects, but always validate authorization again at the actual data-fetching layer.
    • Add generateMetadata to every data-driven, shareable page — it's cheap and has real, direct SEO/social impact.
