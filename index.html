    01. Execution Context & the Call Stack

    MID

    Explanation. Every time JS runs code, it does so inside an execution context: a wrapper holding the currently executing code, its variable environment, its scope chain, and the value of this. There are three kinds — the single Global Execution Context created when the file loads, one Function Execution Context per function call, and Eval (rarely relevant). Contexts are pushed onto the call stack when a function is invoked and popped off when it returns. Each context is built in two phases: a creation phase (hoisting — variable/function declarations are registered, this is bound, the scope chain is set up) and an execution phase (code runs line by line).

    The call stack is strictly LIFO (last-in, first-out) and single-threaded — JS can only do one thing at a time on it. That's the root reason async code needs the event loop instead of just blocking.

    Why interviewers ask it

    It's the fastest way to tell if a candidate has a real mental model of JS or just memorized syntax. It's also the foundation for explaining hoisting, closures, this, and the event loop — so interviewers use it as a gate before going deeper.

    Real-world use case

    Reading a stack trace during a production incident is reading the call stack. When you debug a Laravel queue job or a Node service that throws deep inside a promise chain, the trace tells you exactly which contexts were stacked when it failed — recursive functions blowing the stack (RangeError: Maximum call stack size exceeded) is a real bug class you'll hit refactoring recursive tree logic.

    Common mistakes
    • Saying "JavaScript is single-threaded" without being able to explain what is single-threaded (the call stack / main thread) vs what isn't (Web APIs, libuv thread pool in Node).
    • Confusing the call stack with the event loop's queues — the stack executes synchronous code only; callbacks wait until the stack is empty.
    • Not knowing that a stack overflow is a real, common bug from unbounded recursion.
    Interview answer
    "JS runs on a single call stack. When a function is called, a new execution context is pushed on top; when it returns, it's popped. Each context first goes through a creation phase — where variable and function declarations are hoisted and this/scope are set up — then an execution phase where the code actually runs top to bottom. Because the stack is single-threaded, long synchronous work blocks everything else, which is exactly why we push slow I/O work off to Web APIs / libuv and let the event loop bring the callback back later."
    Follow-up questions
    • What happens on a stack overflow, and how would you fix an unbounded recursive function?
    • Is the call stack the same thing as the event loop?
    • What's in a function execution context besides the code itself?
    // Reading a call stack top-to-bottom
function a() { b(); }
function b() { c(); }
function c() { console.trace('stack here'); }
a();
// Stack (top → bottom) at the trace point: c() → b() → a() → (global)
    Best practices
    • Keep recursive functions bounded, or convert to iterative / tail-call style for large inputs (V8 doesn't reliably optimize tail calls, so prefer iteration for genuinely large loops).
    • Read stack traces bottom-up in most tools to find the origin of an error, top-down to see the immediate throw site.

    02. Scope, Closures, Hoisting & TDZ

    SENIOR

    Explanation. Lexical scope means a variable's accessibility is determined by where it's physically written in the code, not by how the code is called. Every function keeps a reference to the scope it was defined in — that reference is what makes a closure: a function bundled together with references to its surrounding state, which stays alive even after the outer function has returned.

    var is function-scoped and hoisted with its value initialized to undefined. let and const are block-scoped and hoisted too, but they sit in the Temporal Dead Zone (TDZ) from the start of the block until their declaration line — accessing them before that throws a ReferenceError, which is why let/const feel "not hoisted" even though technically they are.

    Why interviewers ask it

    Closures are the single most-tested JS concept because they show up everywhere: React hooks, debounce/throttle, module patterns, private state, event handler factories. If you can't explain why a closure captures a variable by reference (not by value), you'll fail almost every "what does this loop print" question.

    Real-world use case

    Rate-limiting a webhook handler, memoizing an expensive DB lookup, or building a private counter/cache without leaking internal state to the rest of the module — all closures. In a Node/Express service, a common closure pattern is a route factory: a function that takes config and returns a configured handler that "remembers" that config on every request.

    Common mistakes
    • The classic var-in-a-loop bug: all callbacks print the final value of i because var is function-scoped, not per-iteration.
    • Thinking closures copy the variable's value at creation time — they don't, they capture a live reference.
    • Saying let isn't hoisted at all — it is hoisted, it's just unusable until the TDZ ends.
    • Not being able to say why closures can cause memory leaks (holding a reference to a large object/DOM node that never gets garbage collected).
    Interview answer
    "A closure is what you get any time an inner function is defined inside an outer function and keeps access to the outer function's variables after the outer one has returned — because functions in JS carry their lexical scope with them, not just their code. The classic gotcha is var inside a setTimeout loop: since var is function-scoped, all the callbacks share the same variable, so they all log the final value. Switching to let fixes it because let creates a new binding per loop iteration."
    Follow-up questions
    • Fix the classic var-in-loop bug two different ways (one with let, one keeping var).
    • Can a closure cause a memory leak? Give an example.
    • What's the difference between the scope chain and the prototype chain?
    // The classic var-in-loop interview question
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100); // 3, 3, 3
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 100); // 0, 1, 2 — new binding per iteration
}

// A practical closure: private counter, no exposed internal state
function createCounter() {
  let count = 0;
  return { inc: () => ++count, get: () => count };
}
const counter = createCounter();
counter.inc(); counter.inc();
console.log(counter.get()); // 2 — count is unreachable from outside
    Best practices
    • Default to const, use let only when reassignment is needed, avoid var in new code.
    • Null out references to large captured objects when a long-lived closure (e.g. an event listener) no longer needs them.

    03. this, call / apply / bind, Arrow Functions

    MID

    Explanation. this is not determined by where a function is defined — it's determined by how it's called (except for arrow functions). Four binding rules, in priority order: new binding (constructor call → this is the new object), explicit binding (call/apply/bind → this is whatever you pass), implicit binding (obj.method() → this is obj), and default binding (plain function call → this is undefined in strict mode, the global object otherwise). Arrow functions have no own this — they lexically inherit it from the enclosing scope at definition time, which is exactly why they can't be used as object methods that need dynamic this, but are ideal inside callbacks.

    Why interviewers ask it

    Losing this is one of the most common real bugs in class-based or callback-heavy code, and testing it reveals whether a candidate actually understands binding rules or just avoids the problem by pattern-matching "always use arrow functions."

    Real-world use case

    In a React class component (or a Node event emitter class), a method passed as a callback — button.addEventListener('click', this.handleClick) — loses its this unless it's bound in the constructor or defined as an arrow class field. This exact bug shows up constantly in Express route handlers passed as references and in Laravel-adjacent JS frontends wiring up class-based widgets.

    Common mistakes
    • Using an arrow function for an object method that needs this to refer to the object.
    • Not knowing bind returns a new function rather than mutating the original.
    • Forgetting that call and apply differ only in how arguments are passed (list vs array) — conflating them.
    Interview answer
    "this is call-site dependent, not definition-site dependent, for normal functions — so the same function can have a different this depending on how you invoke it. Arrow functions break that rule on purpose: they don't have their own this, they just close over whatever this was in the surrounding scope. That's why arrow functions are great for callbacks inside a method — you get the outer this for free — but bad as the method itself, because then there's no dynamic this to inherit."
    Follow-up questions
    • What does this equal inside a regular function called with no context, in strict vs non-strict mode?
    • Why can't you bind an arrow function's this?
    • How would you fix a class method losing this when passed as a callback — name two ways.
    const user = {
  name: 'Ali',
  greetRegular() { return `Hi, ${this.name}`; },       // this = user
  greetArrow: () => `Hi, ${this.name}`            // this = outer scope, NOT user
};

function introduce(greeting) { return `${greeting}, I'm ${this.name}`; }
introduce.call(user, 'Hello');     // explicit binding, args passed individually
introduce.apply(user, ['Hello']);  // same, args as array
const bound = introduce.bind(user);  // returns a NEW function, permanently bound
bound('Hey');
    Best practices
    • Use arrow class fields (handleClick = () => {...}) instead of manual bind calls in constructors for cleaner React/DOM class code.
    • Never mix arrow functions with call/apply/bind expecting them to change this — they'll silently ignore it.

    04. Objects, Prototypes & Value vs Reference

    MID

    Explanation. Primitives (string, number, boolean, null, undefined, symbol, bigint) are copied by value — assigning or passing one creates an independent copy. Objects (including arrays and functions) are copied by reference — the variable holds a pointer to the object in memory, so two variables can point at the same object and mutating through one affects the other. This is not "pass by reference" in the classical sense though — JS is always pass-by-value, it's just that for objects the value being passed is a reference.

    Every object has an internal [[Prototype]] link (accessible via Object.getPrototypeOf or the legacy __proto__) forming the prototype chain — when you access a property that doesn't exist on the object itself, JS walks up the chain until it finds it or hits null. This is how inheritance works in JS, both via class syntax (syntactic sugar over prototypes) and via Object.create.

    Why interviewers ask it

    Mutation bugs from shared references are one of the most common real-world sources of subtle production bugs — especially in React state and in backend code that mutates a shared config object. This topic also gates whether you understand what class actually compiles to.

    Real-world use case

    Passing a Laravel API response object into multiple functions in a Node BFF layer and having one function accidentally mutate a field that another function then reads — extremely common bug in codebases without immutability discipline. Also directly relevant to React: mutating state directly instead of creating a new object is a top-tier React bug that traces straight back to this concept.

    Common mistakes
    • Believing const obj = {} makes the object immutable — it only prevents reassigning the variable, not mutating its contents.
    • Shallow-copying ({...obj} or Object.assign) when a deep copy was needed, leaving nested objects still shared.
    • Confusing "JS passes objects by reference" with true pass-by-reference (where reassigning the parameter inside the function would affect the caller's variable — it doesn't in JS).
    Interview answer
    "Primitives copy by value, objects copy by reference — but really JS is always pass-by-value; for objects, the value that gets copied is the reference itself. So two variables can point at the same object, and mutating through one is visible through the other, but reassigning one variable to a brand-new object doesn't affect the other. For copying, I reach for a shallow copy with spread when the object is flat, and structuredClone or a deep-clone utility when it's nested and I need full independence."
    Follow-up questions
    • Write a function that mutates an object argument vs one that reassigns it — explain why the caller sees different results.
    • What's the difference between Object.freeze and const?
    • How does class relate to prototypes under the hood?
    function mutate(o) { o.name = 'changed'; }     // mutates the shared object — caller sees it
function reassign(o) { o = { name: 'new' }; } // only rebinds local param — caller unaffected

// Prototype chain in practice
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return `${this.name} makes a sound`; };
const dog = new Animal('Rex');
dog.speak();  // found on Animal.prototype via the chain, not on dog itself
    Best practices
    • Treat state as immutable by default — return new objects/arrays rather than mutating, especially in React and Redux-style state.
    • Use structuredClone(obj) (modern runtimes) for real deep clones instead of the old JSON.parse(JSON.stringify()) hack, which silently drops functions, Date objects, and undefined.

    05. Event Loop, Microtasks & Macrotasks

    SENIOR

    Explanation. JS has one call stack but async work is handled by the runtime (browser Web APIs or Node's libuv) outside that stack. When async work finishes, its callback doesn't run immediately — it's placed in a queue, and the event loop only pulls from that queue once the call stack is completely empty. There are two queue types with different priority: microtasks (Promise callbacks, queueMicrotask, MutationObserver) and macrotasks (setTimeout, setInterval, I/O, UI rendering). After every single macrotask, the event loop drains the entire microtask queue before doing anything else — including before the next macrotask and before a repaint.

    Why interviewers ask it

    This is the single highest-signal JS question for backend/Node candidates specifically, because it directly predicts whether you'll write code with race conditions, starved event loops, or wrong assumptions about ordering in async pipelines (exactly the kind of async retry/webhook logic you've built).

    Real-world use case

    Understanding this is exactly what let you reason correctly about async retry logic for a payment webhook: whether a retry scheduled via setTimeout will fire before or after pending promise resolutions matters for correctness, and getting the ordering wrong causes intermittent, hard-to-reproduce bugs in production that only show up under load.

    Common mistakes
    • Predicting setTimeout(fn, 0) runs "immediately" — it still goes through the macrotask queue and always runs after all pending microtasks and current synchronous code.
    • Not knowing that a chain of many .then() calls can starve macrotasks/rendering if microtasks keep queuing more microtasks.
    • Assuming async/await is "not really" using promises under the hood — it's syntactic sugar over the same microtask mechanism.
    Interview answer
    "The call stack runs synchronous code. Anything async — timers, network, promise resolutions — gets handed off, and when it's ready, its callback goes into either the microtask queue or the macrotask queue. The event loop's rule is: run the current synchronous code to completion, then fully drain the microtask queue, then take exactly one task from the macrotask queue, then drain microtasks again, repeat. That's why a Promise always resolves before a setTimeout(fn, 0) even though both look 'immediate.'"
    Follow-up questions
    • Predict the exact console.log order for a mix of sync code, setTimeout, and .then() calls (classic whiteboard question).
    • Can microtasks starve the event loop? How?
    • How does Node's event loop differ from the browser's (phases: timers, poll, check, etc.)?
    console.log('1 sync');
setTimeout(() => console.log('4 macrotask'), 0);
Promise.resolve().then(() => console.log('3 microtask'));
console.log('2 sync');
// Output order: 1 sync, 2 sync, 3 microtask, 4 macrotask
// — all sync code first, then the full microtask queue, THEN the macrotask.
    Best practices
    • Never assume timer delays are exact — setTimeout(fn, 0) means "as soon as possible after current work," not "now."
    • Break up very long synchronous loops (or long chains of queued microtasks) with a macrotask yield point when responsiveness matters.

    06. Promises, async/await & Combinators

    MID

    Explanation. A Promise is a state machine with three states — pending, fulfilled, rejected — that's guaranteed to settle exactly once. async/await is syntax sugar that lets promise-based code read like synchronous code; an async function always returns a Promise, and await pauses execution of that function (not the whole program) until the awaited promise settles. The four combinators cover distinct real needs: Promise.all (fail-fast, all-or-nothing — rejects as soon as one rejects), Promise.allSettled (waits for every promise regardless of outcome, returns status for each), Promise.race (settles as soon as the first promise settles, win or lose), Promise.any (settles as soon as the first one fulfills, ignoring rejections unless all reject).

    Why interviewers ask it

    Almost every real backend task — calling a payment gateway, hitting multiple APIs in parallel, retrying a flaky webhook — needs correct combinator choice and correct error handling. Picking the wrong combinator (e.g. Promise.all when you actually want allSettled) is a real production bug pattern, not a theoretical one.

    Real-world use case

    Exactly your Yardstik background-check integration: calling multiple downstream services (background check provider + internal DB write + notification dispatch) where you want all of them attempted even if one fails is allSettled; calling several required upstream services where any single failure should abort the whole operation is all.

    Common mistakes
    • Using Promise.all for independent operations where a partial failure should still be reported — silently loses the results of the promises that succeeded.
    • Forgetting to wrap await in try/catch, letting a rejected promise become an unhandled rejection that crashes a Node process.
    • Awaiting promises sequentially in a loop when they're independent and could run in parallel with Promise.all — a common performance bug.
    • Not understanding that an async function's thrown error becomes a rejected promise, not a synchronous throw the caller can catch without await or .catch().
    Interview answer
    "I reach for Promise.all when every call is required and I want fail-fast behavior, allSettled when I need the outcome of every call regardless of individual failures — like fanning out notifications where one failing shouldn't block the others — race for timeout patterns like racing a fetch against a timer, and any for redundancy, like hitting multiple mirrors and taking whichever responds first. For error handling I always wrap awaited calls in try/catch at the boundary where I can actually do something useful with the failure, rather than letting it bubble as an unhandled rejection."
    Follow-up questions
    • You need to call 3 independent APIs and want the response as soon as any succeeds, but only truly fail if all 3 fail. Which combinator, and why?
    • What happens to the other promises in Promise.race after one settles — do they get cancelled?
    • How would you add a timeout to a fetch call that has no built-in timeout?
    // Parallel + partial-failure-tolerant fan-out (webhook-style)
async function notifyAll(channels) {
  const results = await Promise.allSettled(
    channels.map(ch => sendNotification(ch))
  );
  const failed = results.filter(r => r.status === 'rejected');
  if (failed.length) logFailures(failed); // none of them blocked the successful ones
  return results;
}

// Timeout pattern using race
const withTimeout = (promise, ms) => Promise.race([
  promise,
  new Promise((_, reject) => setTimeout(() => reject(new Error('timeout')), ms))
]);
    Best practices
    • Always attach error handling at the point where you can act on the failure — don't let rejections bubble silently.
    • Run independent async work with Promise.all/allSettled instead of sequential await in a loop, unless later calls genuinely depend on earlier results.

    07. ES6+ Essentials: Destructuring, Spread/Rest, Map/Set, Optional Chaining

    MID

    Explanation. Destructuring unpacks values from arrays/objects into variables, including default values and renaming. Spread (...) expands an iterable into individual elements (array copy, function args, object merge); rest uses the same syntax to collect remaining items into an array/object — context (which side of an assignment) tells them apart. Map is a key-value collection where keys can be any type (unlike plain objects, which coerce keys to strings) and preserves insertion order; Set stores unique values. WeakMap/WeakSet hold their keys/values weakly, meaning entries can be garbage-collected if there's no other reference — useful for metadata attached to objects without preventing cleanup. Optional chaining (?.) short-circuits to undefined instead of throwing when accessing a property on null/undefined; nullish coalescing (??) provides a fallback only for null/undefined, unlike || which also triggers on 0, '', and false.

    Why interviewers ask it

    These are the day-to-day fluency markers — they're quick to test and reveal whether you write modern, idiomatic JS or code that still looks like 2013. For a backend dev working with deeply nested API/JSON responses, optional chaining and nullish coalescing specifically test defensive coding habits.

    Real-world use case

    Parsing a third-party API response (like a payment gateway or background-check provider callback) where nested fields may or may not exist is exactly where ?. and ?? prevent a whole category of "Cannot read property of undefined" production crashes.

    Common mistakes
    • Using || for defaults when 0 or '' are valid values — silently overwrites legitimate falsy data. This is a real bug, e.g. a discount of 0% getting replaced by a fallback.
    • Not knowing Object.keys() on a plain object with numeric-like keys coerces them to strings, while Map keys keep their original type.
    • Deep-merging objects with spread, assuming it's a deep clone — spread is shallow only.
    Interview answer
    "I default to nullish coalescing over || whenever 0, empty string, or false could be legitimate values — a discount percentage or a boolean flag are the classic examples where || silently does the wrong thing. For nested API responses I chain optional chaining with a nullish fallback: data?.user?.address?.city ?? 'Unknown' — that's safe against any level of the chain being missing without a pile of manual null checks."
    Follow-up questions
    • When would you use a Map instead of a plain object?
    • What's the difference between spread-copying an object and structuredClone?
    • Why does WeakMap not have a .size property or support iteration?
    // || vs ?? — a real bug class
const discount = 0;
discount || 10;   // 10 — WRONG, discount of 0 got overwritten
discount ?? 10;   // 0  — correct, 0 is a valid value, not "missing"

// Safe nested access on an unpredictable API payload
const city = response?.data?.user?.address?.city ?? 'Unknown';

// Destructuring + rest in one line, common in Express handlers
const { id, ...rest } = req.body;

// Map with non-string keys — not possible with plain objects
const cache = new Map();
cache.set(userObject, { lastSeen: Date.now() });
    Best practices
    • Prefer ?? over || for defaults unless you specifically want every falsy value replaced.
    • Reach for Map/Set when keys aren't strings, order matters, or you need frequent add/delete — they're faster than object shims for that.

    08. Higher-Order Functions: Debounce, Throttle, Memoize, Curry

    SENIOR

    Explanation. A higher-order function either takes a function as an argument or returns one. Debounce delays invoking a function until a pause of silence (e.g. no new calls for N ms) — used for search-as-you-type or resize handlers, where you only care about the final state. Throttle guarantees a function runs at most once per fixed time window regardless of how many times it's triggered — used for scroll handlers or rate-limiting, where you want regular updates, not just the final one. Memoization caches a pure function's return value keyed by its arguments so repeated calls with the same input skip recomputation. Currying transforms a function of multiple arguments into a sequence of functions each taking one argument, enabling partial application.

    Why interviewers ask it

    These are "write it live" questions — they combine closures, timers, and higher-order function composition in one exercise, so they're a favorite for testing whether you can actually write JS under pressure, not just talk about it.

    Real-world use case

    Debounce on a search input hitting your Laravel API so you're not firing a request per keystroke; throttle on a scroll-triggered infinite-load list; memoization on an expensive pure calculation (like formatting/aggregating a large dataset) that gets called repeatedly with the same input inside a render loop.

    Common mistakes
    • Mixing up debounce and throttle conceptually — debounce waits for quiet, throttle enforces a steady rate.
    • Writing debounce/throttle without clearing the previous timer, causing memory leaks or duplicate calls.
    • Memoizing a function with object arguments using a naive cache key (like the object reference) that never matches on subsequent calls with an equivalent-but-different object.
    Interview answer
    "Debounce is for 'wait until they've stopped' — search inputs, resize handlers — I only care about the final call in a burst. Throttle is for 'no more than once every X ms' — scroll handlers, drag events — I want regular sampling throughout, not just the end. I always implement both with a closure holding the timer/last-call reference, since that state needs to persist across calls without leaking into global scope."
    Follow-up questions
    • Implement debounce from scratch, live, in under 2 minutes.
    • How would you memoize a function that takes an object argument?
    • What's a real bug you'd get from forgetting to clear a timer in debounce?
    function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

function throttle(fn, limit) {
  let inCooldown = false;
  return (...args) => {
    if (inCooldown) return;
    fn(...args);
    inCooldown = true;
    setTimeout(() => inCooldown = false, limit);
  };
}

function memoize(fn) {
  const cache = new Map();
  return (...args) => {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}
    Best practices
    • Always clear the previous timer in debounce to prevent stacked, delayed calls.
    • Only memoize pure functions — memoizing something with side effects or non-deterministic output causes stale/incorrect results.

    09. Error Handling & Core Design Patterns

    MID

    Explanation. try/catch/finally handles synchronous errors and (with await) rejected promises; finally always runs regardless of outcome, useful for cleanup like closing a connection. Custom errors extend the built-in Error class to carry structured, typed information (an error code, HTTP status, retryable flag) instead of forcing callers to parse a message string. Of the classic design patterns, four show up constantly in JS interviews: Singleton (exactly one instance, e.g. a single DB connection pool or config object), Factory (a function that creates objects without exposing the exact class/constructor logic to the caller), Observer (subscribers register to be notified of events — the foundation of Node's EventEmitter and DOM events), and the Module pattern (using closures/IIFEs to expose a public API while keeping implementation details private).

    Why interviewers ask it

    Error handling quality is one of the clearest signals of production experience — junior code tends to either swallow errors silently or crash ungracefully, while senior code distinguishes between expected/operational errors (retry, return a clean response) and true bugs (log, alert, fail loudly). Design patterns test whether you can name and apply structure you've already been using instinctively.

    Real-world use case

    Your webhook retry logic needed exactly this distinction: a custom RetryableError vs a PermanentError class lets the retry loop decide programmatically whether to requeue or dead-letter a failed job, instead of parsing error strings — a pattern that maps directly onto Laravel's own exception hierarchy.

    Common mistakes
    • Catching an error and doing nothing (silent failure) — the single worst error-handling anti-pattern, and a fast red flag in interviews.
    • Throwing plain strings instead of Error objects, losing the stack trace.
    • Not distinguishing operational errors (bad user input, network timeout — expected, handle gracefully) from programmer errors (null pointer, type error — bugs, should surface loudly).
    Interview answer
    "I build a small hierarchy of custom error classes extending Error, each carrying a machine-readable code and whether it's retryable — that lets calling code make a decision programmatically instead of string-matching a message. For patterns, the ones I actually reach for day to day are Factory for creating configured objects without leaking constructor details, and Observer/EventEmitter for decoupling side effects — like triggering a notification after a payment succeeds — from the core business logic that shouldn't need to know who's listening."
    Follow-up questions
    • How do you handle an error inside a Promise chain vs inside async/await?
    • When is a Singleton actually a bad idea?
    • Implement a minimal EventEmitter (on/emit) from scratch.
    class AppError extends Error {
  constructor(message, { code, retryable = false } = {}) {
    super(message);
    this.code = code;
    this.retryable = retryable;
  }
}

try {
  await chargePayment(order);
} catch (err) {
  if (err instanceof AppError && err.retryable) requeue(order);
  else deadLetter(order, err);
} finally {
  await releaseLock(order.id);
}

// Minimal Observer / EventEmitter
class Emitter {
  listeners = {};
  on(event, cb) { (this.listeners[event] ??= []).push(cb); }
  emit(event, ...args) { (this.listeners[event] || []).forEach(cb => cb(...args)); }
}
    Best practices
    • Never write an empty catch block — at minimum log with context; ideally decide retry vs fail.
    • Keep custom error classes small and codes stable — treat them as part of your API contract if they cross service boundaries.

    10. DOM: Manipulation, Events, Delegation & Storage

    MID

    Explanation. The DOM is a tree of nodes representing the page; JS mutates it via APIs like querySelector, createElement, appendChild, and the classList/style APIs. Every DOM mutation that affects layout can trigger reflow (recalculating geometry) and repaint — batching reads and writes matters for performance. Events flow through three phases: capturing (root → target), target, and bubbling (target → root) — most listeners run in the bubbling phase by default. Event delegation exploits bubbling: instead of attaching a listener to every child element, attach one listener to a common parent and inspect event.target — critical for lists with dynamically added items. stopPropagation() halts further bubbling/capturing; preventDefault() stops the browser's default action (like a link navigating or a form submitting) without stopping propagation. For storage: cookies are small (~4KB), sent with every HTTP request, and can be scoped/expired server-side — used for auth tokens/sessions; localStorage persists indefinitely per-origin with no expiry and never leaves the browser; sessionStorage is scoped to a single tab and cleared on tab close.

    Why interviewers ask it

    DOM fundamentals test whether you understand the browser environment beyond a framework's abstractions — React, for instance, is built entirely on delegation (it attaches one listener at the root and simulates bubbling internally), so understanding native event flow explains a lot of React's own behavior.

    Real-world use case

    A dynamically rendered list (search results, notification feed) where new items are added after page load — attaching individual listeners per item either misses new items or leaks memory; delegating to the parent container handles both correctly and performs better at scale. Storage choice matters directly for auth: an access token in localStorage is readable by any injected script (XSS risk), which is why many production setups keep the refresh token in an httpOnly cookie instead.

    Common mistakes
    • Attaching a listener to every list item instead of delegating, then forgetting to re-attach when new items are added dynamically.
    • Confusing stopPropagation (stops bubbling) with preventDefault (stops default browser behavior) — they solve different problems and are often both needed together.
    • Storing sensitive tokens in localStorage without acknowledging the XSS trade-off vs httpOnly cookies.
    • Triggering layout thrashing by interleaving DOM reads and writes in a loop instead of batching them.
    Interview answer
    "For dynamic lists I always use event delegation — one listener on the parent, then check event.target (or closest() to handle clicks on nested elements) — because it survives items being added or removed without re-binding anything. On storage, I default to httpOnly cookies for anything sensitive like auth tokens, since JS can't read them, which closes off a whole XSS attack surface that localStorage leaves open; I'll use localStorage for genuinely non-sensitive UI preferences that should persist across sessions."

document.querySelector('#list').addEventListener('click', (e) => {
  const item = e.target.closest('.list-item');
  if (!item) return; // click wasn't on/inside a list item
  handleItemClick(item.dataset.id);
});

// preventDefault vs stopPropagation — often used together
form.addEventListener('submit', (e) => {
  e.preventDefault();   // stop the actual page navigation/reload
  e.stopPropagation(); // stop it bubbling to a parent listener too
  submitViaFetch(form);
});
    Best practices
    • Default to delegation for any list/collection of interactive elements, especially if items are added/removed dynamically.
    • Batch DOM reads together and writes together (or use requestAnimationFrame) to avoid forced synchronous reflow.
