    01. Node's Event Loop Phases & libuv

    SENIOR

    Explanation. Node's event loop, implemented by libuv, runs through distinct phases each cycle, not just the simplified "microtask vs macrotask" model browsers get taught with: timers (runs due setTimeout/setInterval callbacks), pending callbacks, poll (retrieves new I/O events — the phase where most work happens, including reading files/sockets), check (runs setImmediate callbacks), and close callbacks. Between every phase transition, Node drains the microtask queue (Promises) and Node's own process.nextTick queue — with nextTick callbacks running before Promise microtasks, giving Node a slightly different ordering guarantee than a browser. CPU-bound work (heavy synchronous loops, expensive JSON parsing, image processing) blocks this single thread entirely — I/O is what Node offloads efficiently (via libuv's thread pool for filesystem/DNS, and the OS's async I/O for networking), not general computation.

    Why interviewers ask it

    This tests real Node-specific depth beyond the generic browser event loop model — knowing the phase names and, more importantly, knowing that CPU-bound work still blocks Node despite it being "async by default," which is the root cause of a very real class of production incidents (a slow request handler stalling every other concurrent request).

    Real-world use case

    A synchronous, computationally heavy operation — generating a large PDF report, hashing a big payload, deep JSON transformation — inside a request handler blocks the entire Node process, meaning every other concurrent user's request stalls until it finishes; this is exactly the kind of bug that doesn't show up in local testing with one user but cripples a production API under real concurrent load.

    Common mistakes
    • Assuming Node is "non-blocking" means CPU-bound work is also handled asynchronously — it isn't; only I/O is offloaded, CPU-bound synchronous code still fully blocks the single thread.
    • Confusing setImmediate with setTimeout(fn, 0) — they run in different phases, and their relative ordering actually depends on whether they're scheduled from within an I/O callback or not.
    • Not knowing process.nextTick runs before Promise microtasks and can, if abused recursively, starve the event loop entirely, since it's drained fully before moving to the next phase.
    Interview answer
    "The event loop in Node runs through concrete phases via libuv — timers, I/O callbacks, poll, check for setImmediate, close callbacks — with process.nextTick and Promise microtasks draining fully between each phase transition, nextTick first. The thing that actually matters day to day though is that Node only offloads I/O asynchronously — CPU-bound synchronous work, like a heavy computation or a large synchronous JSON transform, still fully blocks the single thread, stalling every other concurrent request. For genuinely CPU-heavy work I'd move it to a worker thread or a separate queued job rather than running it inline in a request handler."
    Follow-up questions
    • Why can a single slow synchronous operation in one request affect every other concurrent user?
    • What's the difference between setImmediate and setTimeout(fn, 0), and does their ordering ever flip?
    • How would you move genuinely CPU-heavy work off the main thread in Node?
    // CPU-bound work blocks EVERY concurrent request — a real production trap
app.get('/report', (req, res) => {
  const result = expensiveSyncComputation(); // blocks the whole process until done
  res.json(result);
});

// Fix: offload to a worker thread, keep the main thread free for I/O
const { Worker } = require('worker_threads');
app.get('/report', (req, res) => {
  const worker = new Worker('./report-worker.js');
  worker.on('message', result => res.json(result));
});

// process.nextTick runs before Promise microtasks, both before the next phase
Promise.resolve().then(() => console.log('promise microtask'));
process.nextTick(() => console.log('nextTick — runs first'));
    Best practices
    • Move genuinely CPU-intensive work to worker threads or a background job queue — never run it inline in a request handler.
    • Avoid recursive process.nextTick calls that never let the loop move to the next phase — a real starvation risk.

    02. Streams & Buffers

    SENIOR

    Explanation. A Buffer is Node's way of handling raw binary data directly in memory — fixed-size, outside the V8 heap. A stream processes data incrementally in chunks rather than loading an entire resource into memory at once — critical for large files, network responses, or any data too big (or too slow-arriving) to buffer wholly upfront. Node has four stream types: Readable (a source, like a file read or HTTP request body), Writable (a destination, like a file write or HTTP response), Duplex (both, like a TCP socket), and Transform (a duplex stream that modifies data as it passes through, like gzip compression). .pipe() connects streams together and automatically handles backpressure — pausing a fast producer when a slower consumer can't keep up, preventing unbounded memory growth.

    Why interviewers ask it

    Streams are the difference between a service that handles a 10MB file fine and crashes on a 2GB one — this topic tests whether you understand Node's actual memory model for I/O-heavy work, which matters directly for any file upload/download or large data export feature.

    Real-world use case

    Generating a large CSV export of bookings and sending it directly to the HTTP response as a stream, rather than building the entire CSV string in memory first, keeps memory usage flat and constant regardless of whether the export is 100 rows or 500,000 — loading it all into memory first is exactly the kind of thing that works fine in development and then crashes the process in production under real data volume.

    Common mistakes
    • Loading an entire large file into memory (fs.readFileSync) before processing or sending it, when a stream would process it incrementally with constant memory usage regardless of file size.
    • Manually piping data without accounting for backpressure, allowing a fast producer to overwhelm a slow consumer's memory buffer — exactly what .pipe() and proper stream event handling exists to prevent.
    • Not handling stream error events — an unhandled 'error' event on a stream crashes the process, unlike a rejected promise which at least surfaces as an unhandled rejection warning.
    Interview answer
    "Anytime I'm dealing with a file or dataset that could be large or arrives incrementally, I reach for streams instead of loading everything into memory first — a CSV export of bookings piped straight to the HTTP response keeps memory flat whether it's 100 rows or 500,000, where building the full string first would eventually crash the process at scale. The part that's easy to get wrong is backpressure — .pipe() handles it automatically by pausing the readable side when the writable side's buffer is full, but if I'm manually consuming stream events instead of piping, I have to handle that pause/resume myself or risk unbounded memory growth."
    Follow-up questions
    • What is backpressure, and how does .pipe() handle it automatically?
    • What happens if a stream emits an 'error' event with no listener attached?
    • When would you use a Transform stream? Give a concrete example.
    const fs = require('fs');
const zlib = require('zlib');

// Stream a large export straight to the response — constant memory usage
app.get('/export.csv', (req, res) => {
  res.setHeader('Content-Type', 'text/csv');
  const bookingStream = getBookingsCsvStream(); // a Readable stream
  bookingStream.on('error', (err) => { logError(err); res.status(500).end(); });
  bookingStream.pipe(res); // backpressure handled automatically
});

// Piping through a Transform stream (gzip) — data modified as it flows through
fs.createReadStream('large-file.txt')
  .pipe(zlib.createGzip())
  .pipe(fs.createWriteStream('large-file.txt.gz'));
    Best practices
    • Default to streaming for anything large or unbounded (file exports, uploads, large API responses) instead of buffering fully in memory.
    • Always attach an error listener to every stream in a pipeline — an unhandled stream error crashes the process.

    03. Express Middleware & the Request Lifecycle

    MID

    Explanation. Express middleware is a chain of functions, each receiving (req, res, next), that a request passes through in the order they're registered — each middleware can inspect/modify the request or response, end the cycle by sending a response, or call next() to pass control to the next one. Order matters critically: middleware registered earlier runs first, and a middleware that never calls next() or sends a response leaves the request hanging forever. Error-handling middleware is distinguished by having four parameters ((err, req, res, next)) instead of three, and Express routes any error passed to next(err) straight to the nearest one, skipping all normal middleware in between.

    Why interviewers ask it

    Middleware ordering bugs are a very common, sometimes hard-to-spot real issue — like an auth check registered after the routes it was supposed to protect, silently letting unauthenticated requests through — so this tests both syntax knowledge and the mental model of the request as flowing through an ordered pipeline.

    Real-world use case

    A typical production middleware chain — request logging, then CORS, then body parsing, then auth, then rate limiting, then the actual route handler, then a catch-all error handler — mirrors exactly the layered request lifecycle you'd build in any Express or Laravel-style backend, where getting the auth-before-routes ordering wrong is a real, exploitable security bug, not just a style issue.

    Common mistakes
    • Registering an auth middleware after the routes it's meant to protect — Express runs middleware strictly in registration order, so anything above an unprotected route never gets the check applied.
    • Forgetting to call next() in a middleware that doesn't end the response itself, leaving the request hanging until it times out.
    • Writing a normal 3-parameter middleware and expecting it to catch errors — only a 4-parameter function is recognized by Express as error-handling middleware.
    Interview answer
    "I think of Express middleware as a strictly ordered pipeline — each request flows through registered middleware top to bottom, and each one either passes control forward with next() or ends the response. The ordering bug I specifically watch for in code review is auth-related middleware registered after routes it's meant to protect, since anything above that point in the file runs completely unprotected — Express doesn't retroactively apply middleware to routes registered before it. For error handling, I make sure the error-handling middleware — the four-parameter kind — sits at the very end of the chain, and that any async route handler explicitly passes caught errors to next(err) rather than letting them become unhandled rejections."
    Follow-up questions
    • What happens if a middleware neither calls next() nor sends a response?
    • How does Express distinguish error-handling middleware from regular middleware?
    • Where would you put a rate-limiting middleware relative to auth, and why?
    const express = require('express');
const app = express();

app.use(requestLogger);          // 1. logging first
app.use(express.json());          // 2. body parsing
app.use(authenticate);           // 3. auth BEFORE any protected route below

app.get('/bookings', (req, res) => { res.json(req.user.bookings); });

// Error-handling middleware — 4 params, always LAST
app.use((err, req, res, next) => {
  logError(err);
  res.status(500).json({ error: 'Something went wrong' });
});

// Async handler must forward caught errors — Express doesn't auto-catch promise rejections
app.get('/camps/:id', async (req, res, next) => {
  try { res.json(await getCamp(req.params.id)); }
  catch (err) { next(err); }
});
    Best practices
    • Register cross-cutting middleware (logging, parsing, auth) before any route that depends on it — order is not cosmetic.
    • Always forward caught async errors to next(err) rather than letting them become unhandled rejections.

    04. REST API Design: Status Codes, Versioning, Pagination

    MID

    Explanation. REST maps operations onto HTTP methods and status codes rather than inventing custom verbs: GET (read, safe/idempotent), POST (create), PUT (full replace, idempotent), PATCH (partial update), DELETE (remove, idempotent). Status codes communicate outcome precisely: 2xx success (200 OK, 201 Created, 204 No Content), 4xx client error (400 bad request/validation, 401 unauthenticated, 403 unauthorized/forbidden, 404 not found, 409 conflict, 422 unprocessable entity, 429 too many requests), 5xx server error (500 internal error, 503 unavailable). Versioning (URL path /v1/, a header, or content negotiation) lets an API evolve without breaking existing clients. Pagination, filtering, and sorting are typically exposed via query parameters (?page=2&sort=-created_at&status=active) rather than baked into the URL path.

    Why interviewers ask it

    API design consistency is what makes an API pleasant or painful for every consumer forever — this is a fast way to check whether a candidate has internalized real conventions (correct status code per situation, idempotency semantics) versus just returning 200 for everything with an error message buried in the body.

    Real-world use case

    Designing the booking creation endpoint means returning 201 with the created resource (not 200), 409 specifically if the slot was already booked by someone else (a genuine conflict, distinct from generic validation failure), and 422 for a malformed request body — precise enough that the frontend team consuming your API (or a mobile client, or another service) can handle each case programmatically instead of parsing error message strings.

    Common mistakes
    • Returning 200 for everything, including actual errors, with the real status buried in a JSON body — breaks HTTP-level tooling (caching, monitoring, client error handling) that relies on the status code itself.
    • Using PUT for a partial update (should be PATCH) or making POST idempotent when the semantics require each call to genuinely create something new — violates what clients and intermediaries (like retry logic) correctly assume about each verb.
    • Not versioning an API at all, or breaking existing consumers with a non-backward-compatible change under the same version — a real production incident, not a theoretical concern, the moment there's more than one client.
    Interview answer
    "I treat the status code as part of the actual API contract, not decoration — 201 with the created resource on a successful POST, 409 specifically for a real conflict like a double-booked slot versus a generic 422 for a malformed request, 401 versus 403 depending on whether the problem is 'who are you' versus 'you can't do that.' That precision matters because clients — a frontend, a mobile app, another service — can branch on the status code programmatically instead of parsing an error message string. For versioning, I default to a URL path version from day one, even before there's a second client, because retrofitting versioning onto a live API with real consumers is a much harder migration than starting with it."
    Follow-up questions
    • What's the difference between 401 and 403, concretely?
    • Why does idempotency matter for PUT and DELETE specifically?
    • How would you introduce a breaking change to an API that already has real consumers?
    // Precise status codes as part of the actual contract
app.post('/v1/bookings', async (req, res) => {
  const { error } = validateBooking(req.body);
  if (error) return res.status(422).json({ error: error.message });

  const available = await checkAvailability(req.body.campId, req.body.date);
  if (!available) return res.status(409).json({ error: 'Slot already booked' });

  const booking = await createBooking(req.body);
  res.status(201).json(booking); // 201, not 200 — a resource was created
});

// Query-param based pagination/filtering/sorting, not baked into the path
// GET /v1/bookings?page=2&per_page=20&status=confirmed&sort=-created_at
    Best practices
    • Use the status code precisely as part of the contract; don't collapse everything to 200/500.
    • Version from the start of a public/multi-consumer API, even before a second client exists.

    05. Authentication: JWT vs Sessions vs OAuth

    SENIOR

    Explanation. Session-based auth stores session state server-side (in memory, Redis, or a DB), with the client holding only an opaque session ID cookie — the server can instantly invalidate a session, but requires shared/centralized session storage across multiple server instances. JWT-based auth is stateless — the token itself carries the claims (user ID, role, expiry), cryptographically signed so the server can verify it without a database lookup, which scales horizontally without shared session storage, but at the real cost that a JWT can't be instantly revoked before its expiry without adding back some form of server-side state (a blocklist), which partially defeats the "stateless" benefit. OAuth 2.0 is a different problem entirely — it's a delegation protocol for letting a user grant a third-party application limited access to their data on another service (like "Sign in with Google") without sharing their actual password with that third party, via access/refresh tokens exchanged through a defined flow.

    Why interviewers ask it

    Conflating these three is extremely common, and the interviewer is checking whether you understand the actual trade-offs (statelessness vs instant revocation) rather than just "JWT is modern, sessions are old" — a real architectural decision, not a fashion choice.

    Real-world use case

    For an API serving both a web frontend and a mobile app (a common shape for a booking platform), JWTs are the natural fit since they don't require sticky sessions or shared session storage across server instances — but that means building in a short access-token expiry plus a refresh-token rotation flow specifically to bound how long a stolen or compromised token stays valid, since you've given up instant server-side revocation as the trade-off for statelessness.

    Common mistakes
    • Treating JWTs as instantly revocable the same way sessions are — without extra infrastructure (a blocklist, short expiry + refresh rotation), a leaked JWT stays valid until it naturally expires, full stop.
    • Storing sensitive data unencrypted in a JWT payload — JWTs are signed (tamper-evident) but not encrypted by default, so the payload is readable by anyone who has the token, including the end user.
    • Confusing OAuth (delegated third-party authorization) with your own application's primary authentication mechanism — OAuth solves "let this app access your Google data," not "how does my app know who you are" on its own, though OIDC (built on OAuth) does address that specifically.
    Interview answer
    "The real trade-off between sessions and JWTs is statelessness versus revocation control — sessions let me kill access instantly by deleting server-side state, JWTs scale without shared session storage but can't be revoked before expiry without adding that state back in some form, like a short-lived access token paired with a refresh token I can invalidate. So for a JWT setup I keep access tokens short-lived — minutes, not days — specifically to bound the damage window if one leaks. OAuth is solving a different problem entirely — letting a user grant a third-party app limited access to their data elsewhere without handing over their password — it's not a replacement for my own app's session/token strategy, though I'd use an OIDC flow on top of it if I wanted 'Sign in with Google' as an actual authentication method."
    Follow-up questions
    • How would you revoke a JWT before its natural expiry?
    • Is a JWT's payload encrypted? What are the implications of that?
    • What's the difference between OAuth and OIDC?
    // JWT: short-lived access token + longer-lived, revocable refresh token
const accessToken = jwt.sign({ userId, role }, SECRET, { expiresIn: '15m' });
const refreshToken = jwt.sign({ userId }, REFRESH_SECRET, { expiresIn: '7d' });
// refreshToken's ID stored server-side — CAN be revoked/rotated on logout or compromise

// Middleware verifying the access token — no DB lookup needed, that's the "stateless" part
function authenticate(req, res, next) {
  try {
    req.user = jwt.verify(req.headers.authorization?.split(' ')[1], SECRET);
    next();
  } catch { res.status(401).json({ error: 'Invalid or expired token' }); }
}
    Best practices
    • Keep JWT access tokens short-lived; use a revocable refresh token to bound the damage window of a leaked token.
    • Never put sensitive data unencrypted in a JWT payload — treat it as readable, not private.

    06. Rate Limiting & API Security (Helmet, CORS, Validation)

    MID

    Explanation. Rate limiting caps how many requests a client can make in a time window (typically tracked by IP or authenticated user ID in Redis), protecting against abuse, brute-force attacks, and accidental client bugs that hammer an endpoint — responding with 429 Too Many Requests once exceeded. CORS (Cross-Origin Resource Sharing) is a browser-enforced security mechanism — the server explicitly declares (via headers) which origins are allowed to make requests to it; without correct CORS headers, browsers block cross-origin JS requests by default, which is a feature, not a bug to "fix" by allowing all origins carelessly. Helmet is an Express middleware that sets a collection of security-related HTTP headers (preventing clickjacking, disabling MIME-sniffing, enforcing HTTPS, etc.) with sensible defaults in one line. Input validation at the API boundary (checking types, formats, ranges before any business logic runs) is the first line of defense against malformed data and a wide class of injection-style attacks.

    Why interviewers ask it

    These are the concrete, checklist-level security practices every production API needs, and they're highly practical — an interviewer testing "have you actually shipped and hardened a real API" versus only ever run it locally with no adversarial traffic.

    Real-world use case

    Rate limiting the login endpoint specifically (tighter than general API rate limits) is a direct defense against credential-stuffing/brute-force attacks, and CORS configured to allow only your actual frontend's origin (not *) prevents a malicious site from making authenticated requests to your API using a logged-in user's browser session/cookies.

    Common mistakes
    • Setting CORS to allow all origins (*) as a quick fix for a dev-environment CORS error, then shipping that to production — opens the API to being called from any malicious site.
    • Applying one flat rate limit across the whole API instead of tighter limits on sensitive endpoints (login, password reset) where brute-force risk is highest.
    • Trusting client-side validation alone and skipping server-side validation — client-side checks are a UX nicety, not a security boundary, since they're trivially bypassed by anyone calling the API directly.
    Interview answer
    "I treat these as layered, not optional extras bolted on later. CORS gets configured to allow only the actual known frontend origins, never a wildcard in production, since it's specifically there to stop a malicious site from riding a user's authenticated session. Rate limiting isn't one flat number across the API — auth endpoints like login get a much tighter limit given the brute-force risk. And validation always happens server-side regardless of what the client already checks, because client-side validation is a UX convenience, not a security boundary — anyone can call the API directly and skip the frontend entirely."
    Follow-up questions
    • Why is client-side validation not sufficient on its own?
    • What specific risk does allowing CORS from '*' introduce?
    • Why would you rate-limit the login endpoint more tightly than a general read endpoint?
    const helmet = require('helmet');
const cors = require('cors');
const rateLimit = require('express-rate-limit');

app.use(helmet()); // sensible security headers, one line

app.use(cors({ origin: 'https://campvine.com' })); // explicit origin, never '*' in production

// General API limit
app.use(rateLimit({ windowMs: 60_000, max: 100 }));

// Much tighter limit specifically on the login endpoint — brute-force defense
app.post('/login', rateLimit({ windowMs: 60_000, max: 5 }), loginHandler);
    Best practices
    • Set CORS to explicit known origins, never a wildcard, in any production API handling authenticated requests.
    • Apply tighter, endpoint-specific rate limits to sensitive actions (login, password reset, payment) beyond the general API limit.

    07. Caching Strategies & CDNs

    MID

    Explanation. Caching exists at multiple layers, each solving a different problem: HTTP/browser caching (Cache-Control, ETag headers) lets the browser or an intermediary skip re-fetching unchanged resources entirely; a CDN caches static assets (and sometimes full page responses) at edge locations geographically close to users, cutting latency and origin server load; application-level caching (Redis/Memcached) stores computed results or database query output in memory, avoiding repeated expensive work. The three core cache-invalidation strategies: TTL-based (expire after a fixed time — simple, but can serve stale data within the window), event-based/explicit invalidation (clear the specific cache key when the underlying data changes — precise, but requires remembering every write path that should trigger it), and cache-aside (application checks cache first, falls back to the source on a miss and populates the cache — the most common pattern in practice).

    Why interviewers ask it

    Caching is deceptively simple to explain and genuinely hard to get right in practice — "there are only two hard problems in computer science: cache invalidation and naming things" is a cliché precisely because stale-cache bugs are so common and often subtle, showing up as "why is this user seeing old data" incidents.

    Real-world use case

    Serving camp listing images and other static assets through a CDN cuts both latency (served from an edge location near the user) and load on the origin server, while the actual camp availability data — which changes — needs a much shorter, deliberately-invalidated cache since serving stale availability could mean showing a slot as open that's already booked.

    Common mistakes
    • Caching data that changes frequently with a long TTL "for performance," causing users to see meaningfully stale information without realizing it.
    • Relying purely on TTL expiration for data where staleness has real consequences (like availability or pricing), instead of also explicitly invalidating on the specific write path that changes it.
    • Not setting any HTTP caching headers on genuinely static assets, missing free performance wins that require zero application code.
    Interview answer
    "I think about caching layer by layer — CDN and HTTP caching for genuinely static assets, since that's essentially free performance requiring no invalidation logic at all, and application-level caching with Redis for expensive, frequently-read, infrequently-changing computed data. The TTL length and invalidation strategy depend entirely on how much staleness the data can tolerate — a camp's static description can sit behind a long TTL, but availability data needs either a short TTL or, better, explicit invalidation on the exact write path that changes it, like clearing that cache key the moment a booking is confirmed, since serving stale availability there has a real, visible consequence for a user."
    Follow-up questions
    • What's the difference between cache-aside and write-through caching?
    • How would you decide the right TTL for a specific piece of cached data?
    • What's a cache stampede, and how would you prevent one?
    // Cache-aside pattern — the most common application-level caching approach
async function getCamp(id) {
  const cached = await redis.get(`camp:${id}`);
  if (cached) return JSON.parse(cached);

  const camp = await db.query('SELECT * FROM camps WHERE id = ?', [id]);
  await redis.set(`camp:${id}`, JSON.stringify(camp), 'EX', 3600);
  return camp;
}

// Explicit invalidation on the write path — not relying on TTL alone
async function confirmBooking(campId) {
  await db.query('UPDATE camp_slots SET available = available - 1 WHERE camp_id = ?', [campId]);
  await redis.del(`camp:${campId}:availability`); // invalidate immediately, don't wait for TTL
}
    Best practices
    • Match TTL length and invalidation strategy to how much real-world staleness the specific data can tolerate.
    • Use a CDN for static assets by default — it's close to a free performance win.

    08. Webhooks, Message Queues & Background Jobs

    SENIOR

    Explanation. A webhook is a third party pushing an HTTP request to your server when something happens on their end (a payment succeeds, a background check completes) — the inverse of your code polling them for status. A robust webhook handler validates the request's authenticity (typically an HMAC signature check using a shared secret) before trusting the payload, responds quickly (200) to acknowledge receipt, and does the actual processing asynchronously via a message queue rather than inline in the request — since the sender will retry on timeout or non-2xx response, and slow inline processing risks that retry firing while the first attempt is still running. Message queues (RabbitMQ, SQS, or Laravel's own queue drivers) decouple producers from consumers, buffer load spikes, and enable retry-with-backoff for failed processing. The single non-negotiable property for any of this: idempotency — since webhooks and queued jobs can and will be delivered/retried more than once, processing the same event twice must produce the same end state, not a duplicate side effect.

    Why interviewers ask it

    This is directly your demonstrated production experience — webhook signature validation, async processing, and retry-safe idempotent handling are exactly the skills behind the Yardstik background-check integration, and interviewers use this to separate "has read about webhooks" from "has actually built one that survives real-world retries and partial failures."

    Real-world use case

    The background-check webhook handler: verify the HMAC signature against the shared secret first (reject anything that fails), respond 200 immediately once the payload is validated and queued, then process it asynchronously with an idempotency check (has this specific check ID already been marked complete?) so a duplicate delivery of the same webhook event — which providers explicitly guarantee can happen — doesn't double-process it or send a duplicate notification.

    Common mistakes
    • Trusting an incoming webhook payload without verifying its signature — without that check, anyone who discovers the endpoint URL can send fabricated events.
    • Processing a webhook fully inline before responding, risking a timeout that triggers the provider's automatic retry while the first attempt is still mid-processing — a classic cause of duplicate processing.
    • Building a queue consumer without idempotency, assuming "at-least-once delivery" from the queue system means "exactly-once" — most queue systems explicitly guarantee at-least-once, not exactly-once, so duplicates are expected, not an edge case.
    Interview answer
    "For any webhook I build, the non-negotiables are signature verification before trusting the payload at all, responding fast so the provider doesn't time out and retry while I'm still processing, and doing the real work asynchronously through a queue rather than inline in the request. The property that ties all of it together is idempotency — providers explicitly guarantee at-least-once delivery, not exactly-once, so I design every handler assuming the same event could arrive twice, usually with a check against the event's unique ID before applying any state change, so a duplicate delivery is a safe no-op instead of a duplicate charge or notification."
    Follow-up questions
    • Walk through exactly how you'd verify an incoming webhook's authenticity.
    • Why should a webhook handler respond quickly and process asynchronously instead of doing the work inline?
    • How do you make a queued job idempotent in practice — what does the actual check look like?
    const crypto = require('crypto');

app.post('/webhooks/background-check', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-signature'];
  const expected = crypto.createHmac('sha256', WEBHOOK_SECRET).update(req.body).digest('hex');
  if (signature !== expected) return res.status(401).end(); // reject unverified payloads

  const event = JSON.parse(req.body);
  queue.add('process-check', { eventId: event.id, payload: event }); // queue it, don't process inline
  res.status(200).end(); // ack fast — before slow processing happens
});

// Idempotent job processor — safe even if the same event arrives twice
async function processCheck({ eventId, payload }) {
  const alreadyProcessed = await db.exists('processed_events', eventId);
  if (alreadyProcessed) return; // safe no-op on duplicate delivery
  await applyCheckResult(payload);
  await db.insert('processed_events', { eventId });
}
    Best practices
    • Always verify webhook signatures before trusting any payload data.
    • Design every queued/webhook-triggered handler assuming at-least-once delivery — idempotency is not optional.

    09. Microservices vs Monolith & Service Communication

    SENIOR

    Explanation. A monolith deploys the entire application as a single unit — simpler to develop, test, deploy, and reason about locally, with easy in-process function calls and straightforward transactions, but everything scales together and a single team's changes can conflict across the whole codebase as it grows. Microservices split the system into independently deployable services, each owning its own data — allowing independent scaling and deployment and clearer team ownership boundaries, at the real cost of distributed systems complexity: network calls instead of function calls (which can fail, time out, or arrive out of order), no simple cross-service transactions (needing patterns like the Saga pattern instead), and significantly more operational overhead (service discovery, distributed tracing, more complex deployment). Services communicate either synchronously (REST/gRPC — simple, but creates tight coupling and cascading failure risk if a downstream service is slow/down) or asynchronously via a message queue/event bus (looser coupling, better resilience to a downstream outage, but harder to reason about ordering and consistency).

    Why interviewers ask it

    Microservices are often reached for prematurely as an assumed "best practice" rather than a genuine trade-off — this topic tests architectural judgment specifically, whether a candidate can articulate real costs (not just benefits) and make a context-dependent call rather than a one-size-fits-all answer.

    Real-world use case

    A booking platform at the size and team scale you've worked at (sole developer or small team) is a legitimate case for staying a well-structured modular monolith — the DDD/bounded-context work you did on CampVine gets much of microservices' organizational benefit (clear domain boundaries, reduced coupling) without paying the full distributed-systems operational cost, which is exactly the trade-off to be able to articulate rather than assuming microservices are always the "more senior" answer.

    Common mistakes
    • Defaulting to microservices for a new project at small team scale, taking on genuine distributed-systems complexity (network failures, no easy cross-service transactions) before there's an actual organizational or scaling need for it.
    • Splitting services along the wrong boundaries — services that constantly need to call each other synchronously for basic operations indicates the boundary itself is probably wrong, and you've mostly just added network latency and failure modes to what was really one cohesive domain.
    • Not having a real answer for cross-service data consistency once split — assuming a distributed transaction "just works" the way a single-database transaction does.
    Interview answer
    "I don't treat microservices as an automatic upgrade from a monolith — a well-structured, modular monolith with clear domain boundaries, which is exactly what a DDD-style refactor gets you, captures a lot of the organizational benefit without the real cost of distributed systems: network calls that can fail or time out where a function call couldn't, and no simple cross-service transaction, needing something like a Saga pattern instead. I'd reach for actual service separation when there's a genuine driver — independent scaling needs for one part of the system, or separate teams that need to deploy independently without blocking each other — not just because it's presented as the more sophisticated architecture."
    Follow-up questions
    • What's a concrete signal that a monolith should actually be split into services?
    • How do you handle a transaction that spans two services in a microservices architecture?
    • What's the risk of choosing synchronous service-to-service calls for a critical path?
    // Synchronous call — simple, but couples availability to the downstream service being up
const paymentResult = await fetch('https://payment-service/charge', { /* ... */ });

// Asynchronous, event-driven — looser coupling, resilient to payment-service being briefly down
eventBus.publish('booking.created', { bookingId, amount });
// payment-service consumes this independently, on its own retry schedule,
// booking-service doesn't block waiting for a synchronous response
    Best practices
    • Split services along real organizational/scaling drivers, not by default — a modular monolith is a legitimate, often correct choice.
    • Prefer async, event-driven communication for anything that doesn't need an immediate synchronous response, to reduce cascading-failure risk.

    10. WebSockets & Real-Time Systems

    MID

    Explanation. HTTP is inherently request-response — the client always initiates. Real-time features (chat, live notifications, live availability updates) need the server to be able to push data to the client without waiting for a request. WebSockets upgrade an initial HTTP connection into a persistent, full-duplex TCP connection, letting both sides send messages at any time with low overhead per message — the right tool for frequent, bidirectional, low-latency communication. Server-Sent Events (SSE) are simpler: a one-way persistent connection where only the server pushes updates over plain HTTP, sufficient when the client never needs to send anything back over that same channel (like a live notification feed). Long polling (the client repeatedly requests, server holds the connection open until there's new data or a timeout) is the fallback when WebSockets aren't available, at the cost of more overhead and latency.

    Why interviewers ask it

    Choosing the right real-time mechanism (and knowing WebSockets aren't the automatic right answer for every "live" feature) is a practical architecture decision, and this topic also surfaces whether you understand the operational difference — WebSocket connections are stateful and long-lived, meaning they interact very differently with load balancers and horizontal scaling than stateless HTTP requests do.

    Real-world use case

    Live camp availability updates (so a user sees a slot disappear in real time if someone else books it while they're browsing) is well-served by SSE or a lightweight WebSocket channel, since it's fundamentally server-to-client push with no need for the client to send frequent messages back — versus an actual chat feature between an organizer and a guest, which is genuinely bidirectional and a stronger fit for full WebSockets.

    Common mistakes
    • Reaching for WebSockets for a feature that's actually one-directional (server pushing updates only), adding bidirectional complexity that SSE would have handled more simply.
    • Not accounting for WebSocket connections being stateful when scaling horizontally — a message needs to reach the specific server instance holding that user's open connection, which plain round-robin load balancing doesn't guarantee, requiring a pub/sub layer (like Redis) to broadcast across instances.
    • Not planning for reconnection — a dropped WebSocket connection (network blip, server restart) needs explicit client-side reconnect logic; it doesn't recover automatically like a stateless HTTP request retry would.
    Interview answer
    "I pick the real-time mechanism based on directionality first — if it's genuinely server-to-client only, like a live notification feed or availability updates, Server-Sent Events are simpler than a full WebSocket and I don't need the bidirectional complexity. True two-way, frequent communication, like a chat feature, is where WebSockets earn their cost. The operational detail that trips people up scaling either one is that these connections are stateful and long-lived — unlike a stateless HTTP request, a message has to reach the specific server instance holding that user's open connection, so horizontal scaling needs a pub/sub layer like Redis to broadcast messages across instances rather than relying on the load balancer alone."
    Follow-up questions
    • When would SSE be a better fit than a full WebSocket connection?
    • Why does horizontally scaling a WebSocket server need a pub/sub layer like Redis?
    • How would you handle a dropped WebSocket connection on the client?
    // Server-Sent Events — simple one-way push, plain HTTP, no upgrade needed
app.get('/availability-stream', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  const unsubscribe = onAvailabilityChange(update => {
    res.write(`data: ${JSON.stringify(update)}\n\n`);
  });
  req.on('close', unsubscribe);
});

// WebSocket + Redis pub/sub — required for horizontal scaling across instances
io.on('connection', (socket) => {
  socket.join(`camp:${socket.handshake.query.campId}`);
});
// Any server instance publishes; Redis broadcasts to whichever instance
// actually holds the relevant client's socket connection
redisSubscriber.on('message', (channel, message) => {
  io.to(channel).emit('update', JSON.parse(message));
});
    Best practices
    • Choose SSE over WebSockets for purely one-directional server-push features — simpler infrastructure, same result.
    • Plan for horizontal scaling (pub/sub broadcast) and client reconnection logic from the start, not as an afterthought once real traffic hits.
